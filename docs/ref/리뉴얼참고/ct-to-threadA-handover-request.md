## 요청 1: 파이프라인 기록

### 1-1. 전체 흐름

현재 staged pipeline의 단계별:
- 각 단계의 역할, 입력, 출력
- 스크립트 경로
- 단계 간 의존성
- 병렬 실행 구조 (`run-parallel-case-batch.cjs`)

### 1-2. 핵심 스크립트 & lib

`scripts/` 와 `scripts/lib/` 아래 파일들:
- 각 파일의 역할 한 줄 설명
- 어떤 파일이 핵심이고 어떤 파일이 레거시인지
- 수정하면 위험한 파일 표시

### 1-3. validator 전체

각 validator의:
- 무엇을 체크하는지
- PASS/FAIL 기준
- 알려진 한계 (validator PASS이지만 실제 품질은 미달인 경우)

### 1-4. manifest / 설정

- `refined/manifest.json` — 현재 12건 상태, 어떤 기준으로 축소했는지
- `scripts/manifests/*.json` — 각 매니페스트의 용도
- `caseLoader.ts` — USE_REFINED_ONLY, SCRIPTED_SET 등 현재 설정

## 요청 2: 생성기의 능력과 한계

솔직하게 기록해 주세요. 새 스레드들이 착각하지 않도록.

### 2-1. 잘 하는 것
- RuntimeCaseData 구조 생성 (evidence chain, lieConfig, requires 등)
- ScriptedText 골격 생성 (6채널, 키 매칭, variant 수)
- validator 기반 구조 검증
- 병렬 배치 실행

### 2-2. 못 하는 것 / 현재 한계
- 대사 품질 — 템플릿 반복, 기계적 scaffold
- 증인 증언 — 메타 서술 ("증인은 ~합니다")
- 한국어 자연스러움 — JSON 필드명 노출, 공문체
- 세션-스토리 fit 판단 — 생성기가 판단할 영역이 아님

### 2-3. 외부 대사 주입 가능성

**이것이 가장 중요합니다.** 새 Thread W(대사 작성 전담)가 만든 대사를 생성기에 넣을 수 있는지:

- 현재 ScriptedText에 외부 작성 대사를 주입하는 경로가 있는지
- 있다면 입력 포맷은 어떤 형태인지
- 없다면 어떤 구조를 추가해야 하는지
- 생성기가 "구조만 만들고 대사는 외부에서 받는" 모드가 가능한지
