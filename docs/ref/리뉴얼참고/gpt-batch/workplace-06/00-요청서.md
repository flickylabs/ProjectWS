# V3 배치 요청서 — GPT

## 프로젝트 배경

**Solomon**은 LLM 기반 법정 심문 게임입니다. 플레이어는 재판관으로서 양측 당사자(AI NPC)를 심문하여 진실을 밝힙니다.

### 아키텍처
- **엔진이 의미 결정**, LLM은 연기만. 매 턴 `ResponseBlueprint` 생성 → atom 선택 → LLM 1~3문장 응답.
- NPC는 lieState 상태 머신(S0 완전부정 → S5 시인)을 따름.
- **ClaimAtom**: 의미 단위 (사실+슬롯). 기존 publicClaim 문장을 원자로 분해한 것.
- **Slot**: 표현 변형 (exact: "2,800,000원", neutral: "해당 금액", judgeRef: "장모님")
- 질문: fact_pursuit(사실추궁), motive_search(동기탐색), empathy_approach(공감접근), evidence_present(증거제시)

### 이번 요청에서 만들 것
첨부된 사건 원본 JSON을 읽고, 아래 **4가지 데이터**를 한 번에 생성합니다.

## ⚠️ 필수 규칙: ID 네이밍 & 필드명

> **아래 규칙은 절대적**입니다. 예시와 다른 형식을 사용하지 마세요.
> 엔진이 정확한 문자열 매칭으로 데이터를 조회하므로, 패턴이 1글자라도 다르면 런타임 에러가 발생합니다.

### ID 공통 규칙
- **구분자는 반드시 콜론(`:`)** — 점(`.`), 언더스코어(`_`), 하이픈(`-`)을 구분자로 쓰지 마세요
- **caseId 접두사에 하이픈 없음**: `family02`, `friend05` (O) / `family-02`, `friend-05` (X)
- **state는 소문자**: `s0`, `s1`, `s2` (O) / `S0`, `S1` (X)  (단, `applicableStates` 배열값과 `unlockedAtState` 값은 대문자 `"S0"`, `"S2"` 사용)

### slots 필드 키 규칙
- 반드시 `exact`, `neutral` 키를 포함 (필수)
- 선택 키: `rounded`, `dateExact`, `period`, `fullName`, `judgeRef`
- **다른 이름 금지**: `target`, `fallback`, `cropped`, `original` 등 사용하지 마세요

## 요청 A: ClaimPolicy V2 (atom 기반)

각 파티 × 쟁점 × S0~S5 state마다 ClaimPolicy를 생성합니다.
핵심은 `claimAtoms` 배열 — publicClaim 문장을 원자로 분해.

### atom 분해 규칙
1. publicClaim의 각 문장을 1~2개 atom으로 분해
2. 숫자/날짜/인명은 반드시 slot으로 분리
3. 태그: act, timeline, motive, emotion, rule, evidence, context, identity, responsibility, relationship, counter, self_justification, harm, fear, shame, privacy, admission, denial, uncertainty, quote, threshold, beneficiary, institution, legacy_sentence (24종)
4. S0~S1: deny/hedge용 (핵심 회피), S2~S3: partial/blame (부분 인정+전가), S4~S5: emotional/confess (시인)
5. privateKnowledge → 고 state에서 해금되는 atom의 factText 소재
6. 각 atom에 stanceHints, usableInSubActions 부여

### ClaimAtom ID 패턴
```
{{CASE}}:{{PARTY}}:{{DISPUTE}}:{{TAG}}:{{INDEX}}
```
- 예: `spouse01:a:d-1:act:0`, `family02:b:d-3:motive:1`
- `{{CASE}}`: caseId에서 하이픈 제거 (`family-02` → `family02`)
- `{{PARTY}}`: `a` 또는 `b`
- `{{DISPUTE}}`: `d-1`, `d-2` 등 (원본 그대로)
- `{{TAG}}`: atom의 대표 태그 1개 (`act`, `timeline`, `motive` 등)
- `{{INDEX}}`: 같은 dispute+state 내 순번 (`0`, `1`, `2`...)

### tellPool ID 패턴
```
{{CASE}}:{{PARTY}}:tell:{{TELL_NAME}}
```
- 예: `family02:a:tell:hierarchy_strike`
- 요청 B의 Tell id와 **반드시 동일**해야 함

### 출력 구조
```json
{
  "caseId": "{{CASE_ID}}",
  "claimPolicies": {
    "a": {
      "d-1": {
        "S0": {
          "disputeId": "d-1", "state": "S0",
          "publicClaim": ["원문 유지"],
          "privateKnowledge": ["원문 유지"],
          "suppressions": ["원문 유지"],
          "emotionalLeakRisk": "low",
          "tellPool": ["family02:a:tell:hierarchy_strike"],
          "claimAtoms": [
            { "id": "family02:a:d-1:act:0", "factText": "...", "tags": [...], "slots": {...}, "stanceHints": [...], "usableInSubActions": [...] }
          ]
        }
      }
    }
  }
}
```

### atom 예시 (spouse-01에서 발췌)
```json
{
  "id": "spouse01:a:d-1:act:0",
  "factText": "공동계좌에서 이체가 있었다는 사실",
  "tags": ["act", "timeline"],
  "slots": {
    "amount": { "exact": "2,800,000원", "rounded": "280만원", "neutral": "해당 금액" },
    "time": { "dateExact": "9월 20일", "period": "추석 연휴 직전", "neutral": "그 시점" }
  },
  "stanceHints": ["deny", "hedge"],
  "usableInSubActions": ["fact_pursuit", "evidence_present"]
}
```

## 요청 B: ExecutableVerbalTell + BeatScript

### Tell (파티당 3개 = 6개)
사건 원본의 `speechStyle`과 `archetype`을 반영한 말버릇 규칙.

### Tell ID 패턴
```
{{CASE}}:{{PARTY}}:tell:{{TELL_NAME}}
```
- 예: `family02:a:tell:hierarchy_strike`, `family02:b:tell:delay_buffer`
- 구분자는 반드시 콜론(`:`)
- `{{TELL_NAME}}`: snake_case (예: `accusation_loop`, `soft_minimizing`)

```json
{
  "id": "family02:a:tell:hierarchy_strike",
  "appliesWhen": ["lying", "cornered", "emotional", "avoiding", "shame", "hurt", "defensive"],
  "lexicalHooks": ["어휘1", "어휘2"],
  "sentenceShape": "number_first|question_end|enumeration|echo_repeat|conditional|self_reference",
  "cadence": "every_turn|once_every_2_turns|max_once_per_turn|on_trigger_only",
  "originalPattern": "이 말버릇의 설명"
}
```

### Beat (36개)
쟁점별 deny/hedge/partial/blame/confession/evidence_hit × 해당 state.

### BeatScript ID 패턴 (필수 — 생략 금지)
```
{{CASE}}:beat:{{PARTY}}:{{DISPUTE}}:{{BEAT_TYPE}}
```
- 예: `family02:beat:a:d-1:deny`, `family02:beat:b:d-3:confession`
- **`id` 필드는 반드시 포함**해야 합니다

```json
{
  "id": "family02:beat:a:d-1:deny",
  "caseId": "{{CASE_ID}}",
  "party": "a",
  "disputeId": "d-1",
  "beatType": "deny",
  "line": "1~3문장, 캐릭터 말투 반영",
  "behaviorHint": "행동/표정 연출 지시",
  "applicableStates": ["S0", "S1"],
  "afterEvidence": null
}
```

**beatType 허용값** (6종만): `deny`, `hedge`, `partial`, `blame`, `confession`, `evidence_hit`
**applicableStates**: 단일 값 배열 권장 (예: `["S0"]`). 복수 state 지양.

### ⚠️ BeatScript 대사 품질 규칙

> partnership 배치에서 대사 품질 문제가 다수 발견되었습니다. 아래 규칙은 **필수**입니다.

#### beatType별 톤 (엄격 준수)

| beatType | 필수 톤 | 금지 |
|----------|---------|------|
| **deny** | 전면 부인 ("~한 적 없다") | 사실 인정/축소 포함 금지 (hedge와 혼동 금지) |
| **hedge** | 축소/맥락 전환 ("~이긴 하지만") | 전면 부정도, 명시적 시인도 금지 |
| **partial** | 부분 인정 + **반드시 "다만/그래도" 단서** | 단서 없는 전면 인정 금지 (confession과 혼동 금지) |
| **blame** | 인정 + **반드시 "~쪽도..." 상대 책임 전가** | 자기 책임만 인정하면 blame 아님 |
| **confession** | 자기 행위 명시("제가 ~") + **"제 잘못/책임" 표현** + 회피 없는 종결 | 사실 진술만으로 끝내기 금지, 상대 부정으로 시작 금지 |
| **evidence_hit** | 증거 인식 표현 ("~까지 나오면", "부정할 수 없다") | confession과 동일 문장 복사 금지 |

#### lexicalHook 반영 (필수)
- Tell에 정의한 `lexicalHooks`를 해당 party의 beatScript `line`에 **최소 50%** 자연스럽게 녹여야 합니다
- 예: Tell hook이 `["관리규약상", "08시 14분", "사실관계만 보시죠"]`이면, 최소 2개가 beat line 어딘가에 등장해야 함
- hook을 기계적으로 붙이지 말고, 대사 흐름 안에서 자연스럽게 사용

#### 호칭 규칙
- 재판관 앞 상황이므로 **상대를 "{{이름}} 씨"로 호칭** (사건원본 `callTerms.toPartner` 참조)
- 이름만 부르기("서준은", "해나가") 금지 — 반드시 "씨" 또는 원본 callTerms에 맞는 경칭 사용
- 동업/직장 관계에서 "대표", "팀장" 등 직함은 허용

#### 존대/반말 규칙
- **재판관에게 말하는 진술**: 존대 필수
  - cold_logic → 합쇼체 ("~입니다")
  - confrontational → 해요체 강한 ("~잖아요")
  - avoidant → 해요체 부드러운 ("~것 같아요")
  - victim_cosplay → 해요체 호소적 ("~거예요")
- **감정 폭발로 상대에게 직접 던지는 대사**: 반말 허용
- **"~다" 평서체(해라체)**: 전면 사용 금지

#### 대사는 반드시 1인칭 직접 화법
- "제가 ~했습니다", "저는 ~라고 생각해요" (O)
- "본인은 ~라는 입장이다", "이름+은/는 ~고 말한다" (X — 3인칭 서술체 금지)

#### DossierCard 질문 호칭
- 재판관 시점 질문에서 **"{{이름}} 씨,"**로 시작 (각 challenge 첫 질문에 필수)
- "당사자", "피심문인" 사용 금지

## 요청 C: V3 Game Loop

### DossierCard 3개
증거 2개를 묶은 카드. 각 카드에 A 대상 3질문 + B 대상 3질문.

### DossierCard Question ID 패턴
```
dossier-{{N}}.{{PARTY}}.q{{M}}
```
- 예: `dossier-1.a.q1`, `dossier-2.b.q3`
- 이것은 **점(`.`) 구분자**를 사용하는 유일한 예외

### revealAtom ID 패턴 (DossierCard onSuccess + StateUnlockAtom 공통)
```
{{CASE}}:{{PARTY}}:{{DISPUTE}}:unlock:{{STATE}}:{{INDEX}}
```
- 예: `family02:a:d-1:unlock:s2:0`, `family02:b:d-3:unlock:s3:1`
- **DossierCard의 `onSuccess.revealAtom`과 StateUnlockAtom의 `id`는 반드시 동일해야 함**

```json
{
  "id": "dossier-1",
  "name": "카드 이름",
  "description": "1줄 설명",
  "evidenceIds": ["e-1", "e-2"],
  "relatedDisputes": ["d-1", "d-3"],
  "subjectParty": "both",
  "challenges": [
    {
      "targetParty": "a",
      "questions": [
        {
          "id": "dossier-1.a.q1",
          "text": "재판관 말투 질문",
          "attackVector": "identity|context|legality|authenticity",
          "onSuccess": { "blockVector": "identity", "revealAtom": "family02:a:d-1:unlock:s2:0", "lieAdvance": true }
        }
      ]
    }
  ]
}
```

**attackVector 허용값** (4종만): `identity`, `context`, `legality`, `authenticity`

### StateUnlockAtom
a(해당 쟁점) + b(해당 쟁점) × S2/S3/S4/S5 각 1~2개.
**revealAtom ID와 일치해야 함.**

```json
{
  "id": "family02:a:d-1:unlock:s2:0",
  "factText": "새로 드러나는 사실",
  "tags": ["context", "motive"],
  "unlockedAtState": "S2",
  "slots": { "person": { "fullName": "...", "judgeRef": "...", "neutral": "그 사람" } },
  "stanceHints": ["partial", "emotional"]
}
```

### 이벤트 텍스트
- 모순 3~4개, 끼어들기 3개, 감정폭발 양측 각 2개 (총 ~10개)

**events 필드명은 반드시 다음을 사용** (다른 이름 금지):
```json
{
  "contradictions": [
    {
      "id": "{{CASE}}:contradiction:{{INDEX}}",
      "targetParty": "a",
      "trigger": "발동 조건 설명",
      "text": "모순 지적 텍스트",
      "relatedDisputes": ["d-1", "d-4"],
      "evidenceIds": ["e-1"]
    }
  ],
  "interjections": [
    {
      "id": "{{CASE}}:interjection:{{INDEX}}",
      "speaker": "a|b|judge",
      "trigger": "발동 조건",
      "text": "끼어들기 텍스트",
      "relatedDisputes": ["d-2"]
    }
  ],
  "emotionalOutbursts": [
    {
      "id": "{{CASE}}:outburst:{{PARTY}}:{{INDEX}}",
      "party": "a",
      "trigger": "감정 폭발 조건",
      "text": "감정 폭발 텍스트",
      "relatedDisputes": ["d-1"]
    }
  ]
}
```
- 필드명: `trigger` (O) / `cue` (X)
- 필드명: `text` (O) / `line` (X)
- 필드명: `relatedDisputes` (O) / `relatedEvidenceIds` (X)
- **`followUpHint`, `narrationHint` 등 임의 필드 추가 금지**

### TransitionBeat 20~24개
각 파티 × 쟁점 × S1→S2~S4→S5 (4단계).
d-5 같은 쌍방 쟁점은 양측 모두 필요.

### TransitionBeat ID 패턴 (필수 — 생략 금지)
```
{{CASE}}:transition:{{PARTY}}:{{DISPUTE}}:{{FROM_STATE}}_{{TO_STATE}}
```
- 예: `family02:transition:a:d-1:s0_s1`, `family02:transition:b:d-3:s3_s4`

```json
{
  "id": "family02:transition:a:d-1:s0_s1",
  "caseId": "{{CASE_ID}}",
  "party": "a",
  "disputeId": "d-1",
  "fromState": "S0",
  "toState": "S1",
  "trigger": "전이 발동 조건",
  "line": "캐릭터 말투 반영 대사",
  "behaviorHint": "행동/표정 연출 지시"
}
```
- **`id` 필드는 반드시 포함** (엔진이 중복 사용 방지에 활용)
- **`revealAtomIds` 등 임의 필드 추가 금지**

## ⚠️ 최종 체크리스트 (생성 전 반드시 확인)

### ID / 형식
1. [ ] 모든 ID의 구분자가 콜론(`:`)인가? (dossierCard question만 점(`.`) 예외)
2. [ ] caseId 접두사에 하이픈 없는가? (`family02` O / `family-02` X)
3. [ ] beatScript, transitionBeat에 `id` 필드가 있는가?
4. [ ] `onSuccess.revealAtom` ID가 `stateUnlockAtoms` 내 atom `id`와 정확히 일치하는가?
5. [ ] `tellPool` ID가 `executableTells` 내 tell `id`와 정확히 일치하는가?
6. [ ] events 필드명이 `trigger`, `text`, `relatedDisputes`인가? (`cue`, `line`, `relatedEvidenceIds` 아님)
7. [ ] slots 키가 `exact`, `neutral`을 포함하는가? (`target`, `fallback` 아님)
8. [ ] 임의 필드를 추가하지 않았는가? (`followUpHint`, `revealAtomIds`, `narrationHint` 등)

### 대사 품질
9. [ ] Tell의 lexicalHooks가 해당 party의 beat line에 **50% 이상** 반영됐는가?
10. [ ] 모든 confession beat에 "제 잘못/책임" 명시 표현이 있는가?
11. [ ] 모든 blame beat에 "~쪽도..." 상대 전가 표현이 있는가?
12. [ ] 모든 partial beat에 "다만/그래도" 단서가 있는가?
13. [ ] evidence_hit이 confession과 동일 문장이 아닌가?
14. [ ] beat line이 1인칭 직접 화법인가? (3인칭 서술체 "~라는 입장이다" 금지)
15. [ ] 상대 호칭에 "씨" 또는 적절한 경칭이 붙어 있는가?
16. [ ] DossierCard 첫 질문이 "{{이름}} 씨,"로 시작하는가?
