## 1. DossierCard 질문 호칭/어투 통일 기준

### 1-1. 표준 패턴 도출 (PASS 사건 샘플링)

|------|---------------------|---------------------|-----------|
| family-02 | "정미옥 당사자" | "강선우 당사자" | ~입니까, ~보십시오, ~있었습니까 |

### 1-2. 통일 기준

**호칭 규칙:**
- **표준 형식**: `"{{성+이름}} 씨"` (예: "정미옥 씨", "강선우 씨")
- **"~당사자"는 사용하지 않는다.** family-02의 "정미옥 당사자", "강선우 당사자"는 예외적이며, 나머지 PASS 사건 대부분이 "~씨" 패턴을 사용한다.
- 호칭은 **각 challenge의 첫 번째 질문**에서만 사용한다. 두 번째, 세 번째 질문은 이름 없이 시작해도 무방하다.
- "피심문인", "당사자 A/B" 같은 법률 용어는 사용하지 않는다.

**재판관 어투 규칙:**
- 존칭 수준: **합쇼체** (하십시오체) — "~하셨습니까", "~보십시오", "~답하십시오"
- **"재판관:" 프리픽스는 포함하지 않는다.** DossierCard의 question.text는 순수 질문 텍스트만 담는다.
- 문장 종결은 **물음표(?) 또는 마침표(.)** 중 하나로 통일한다.
  - 질문문 → 물음표: "~하셨습니까?"
  - 지시/요구문 → 마침표: "~답해 보십시오."
- 종결 혼용 금지: 한 사건 내에서 같은 유형의 문장에 ?와 .를 섞지 않는다.

**구체적 수정 대상:**
- family-02: "정미옥 당사자" → "정미옥 씨", "강선우 당사자" → "강선우 씨"
  - `dossier-1.a.q1`: "정미옥 당사자, 42초 파일만으로..." → "정미옥 씨, 42초 파일만으로..."
  - `dossier-1.b.q1`: "강선우 당사자, 민지에게 보낸..." → "강선우 씨, 민지에게 보낸..."
  - `dossier-1.b.q3`: "당사자는 왜 그 42초를..." → "강선우 씨는 왜 그 42초를..."

## 2. confession beat 톤 기준

### 2-1. 각 beatType 톤 가이드

| beatType | 핵심 요소 | 필수 톤 | 금지 패턴 |
|----------|-----------|---------|-----------|
| **deny** | 전면 부인 | "~한 적 없다", "~가 아니다" | 사실 인정 포함 금지 |
| **hedge** | 축소/맥락 전환 | "~이긴 하지만", "~는 맞지만 그건..." | 명시적 시인 금지 |
| **partial** | 부분 인정 + 단서 | "~한 건 맞다. 다만..." | 전면 시인 금지, 반드시 단서 첨부 |
| **blame** | 인정 + 상대 책임 전가 | "~한 건 맞지만, ~쪽도..." | 자기 책임만 인정하면 안 됨 |
| **confession** | **명시적 자기 잘못 시인** | 아래 필수 요소 참조 | 상대 부정/비난으로 시작 금지, 사실 진술만으로 끝 금지 |
| **evidence_hit** | 증거 앞 태도 변화 | "~까지 나오면", "~부정 못 하겠다" | 증거 무시 금지 |

### 2-2. confession beat 필수 요소 (체크리스트)

confession beat는 반드시 다음 3가지를 포함해야 한다:

1. **자기 행위 명시**: 자신이 한 구체적 행위를 주어 "제가/내가"로 시작하여 서술
2. **잘못/책임 인정 표현**: 다음 중 하나 이상 포함
   - "제 잘못입니다" / "내 잘못이다"
   - "제 책임입니다" / "내 책임이다"
   - "인정합니다" / "인정해야겠습니다"
3. **회피 없는 종결**: 상대 탓, 변명, 조건부 인정("~이긴 한데...")으로 끝나면 안 됨

**좋은 confession 예시 (PASS 사건들):**
```
# neighbor-02:a:d-1:confession
"네, 제가 캡처 한 장으로 민규 씨를 수상한 사람처럼 보이게 했어요.
 확인보다 제 불안과 인상을 먼저 믿었습니다."
→ ✅ 자기 행위("캡처를 돌림") + 잘못 인정("~처럼 보이게 했어요") + 회피 없음

# neighbor-04:a:d-1:confession
"제가 7시 20분 전에 비워주기로 한 약속을 반복해서 어겼습니다.
 아버지 사정을 앞세워 원상복구를 미룬 건 제 책임입니다."
→ ✅ 자기 행위("약속을 어김") + 책임 인정("제 책임입니다") + 회피 없음

# friend-06:a:d-2:confession
"공개 비난부터 하고 채널 전체 접근권을 끊은 건 내 잘못이다.
 사실 확인보다 체면과 통제권 회복을 앞세웠다."
→ ✅ 자기 행위("공개 비난+접근권 회수") + 잘못 인정("내 잘못이다") + 회피 없음
```

### 2-3. 문제 사건 분석 및 수정 방향

## 3. 존대/반말 통일 기준

### 3-1. archetype별 기본 존대 수준

재판관 앞 상황이므로 **모든 archetype은 존대체를 기본으로 한다.**
archetype에 따른 차이는 존대 안에서의 "격식도"와 "어감"으로 표현한다.

| archetype | 기본 화체 | 허용 표현 범위 | 금지 |
|-----------|-----------|----------------|------|
| **cold_logic** | 합쇼체(하십시오체) | "~입니다", "~습니다", "~봅니다" | ~~"~다" 평서체~~, 반말 |
| **confrontational** | 해요체 (강한) | "~잖아요", "~거예요", "~할 수 있어요?" | ~~"~다" 평서체~~, 반말 |
| **avoidant** | 해요체 (부드러운) | "~인 것 같아요", "~거든요", "~는데요" | 반말 |
| **victim_cosplay** | 해요체 (호소적) | "~했다고요", "~는 거예요", "~잖아요" | ~~"~다" 평서체~~, 반말 |

**핵심 원칙**: 재판관 앞이므로 "~다" 평서체(해라체)는 어떤 archetype에서도 사용하지 않는다.

### 3-2. friend-06 (윤해온, cold_logic) "~다" 체 수정

friend-06의 partyA(윤해온)는 `cold_logic` archetype인데, beatScript 대사가 전반적으로 "~다" 평서체로 작성되어 있다.

**문제 패턴**: "~아니었다", "~한 거다", "~있다", "~내 잘못이다"
**수정 방향**: cold_logic의 합쇼체로 전환 — "~아니었습니다", "~한 겁니다", "~있습니다", "~제 잘못입니다"

단, **friend 관계의 특수성**을 고려하면, 친구 사이라 약간 격식을 낮춘 해요체도 허용할 수 있다.
→ **최종 결정: 해요체로 통일** ("~예요", "~거예요", "~이에요")

**구체적 수정 예시:**

```
# friend06:beat:a:d-2:deny
수정 전: "첫째, 단톡에 올린 건 공개 비난이라기보다 이상 징후 공유였다.
          그때 화면상으론 무단 침범처럼 보여서 먼저 멈춰 세우는 게 맞다고 생각했다."
수정 후: "첫째, 단톡에 올린 건 공개 비난이라기보다 이상 징후 공유였어요.
          그때 화면상으론 무단 침범처럼 보여서 먼저 멈춰 세우는 게 맞다고 생각했어요."

# friend06:beat:a:d-2:confession
수정 전: "공개 비난부터 하고 채널 전체 접근권을 끊은 건 내 잘못이다.
          사실 확인보다 체면과 통제권 회복을 앞세웠다."
수정 후: "공개 비난부터 하고 채널 전체 접근권을 끊은 건 제 잘못이에요.
          사실 확인보다 체면과 통제권 회복을 앞세웠어요."

# friend06:beat:a:d-3:deny
수정 전: "첫째, 다솔에게 공개 수정까지 허용한 적은 없었다.
          내가 준 건 함께 검토하자는 초대에 가까웠지 라이브 게시물을 바꾸라는 뜻은 아니었다."
수정 후: "첫째, 다솔에게 공개 수정까지 허용한 적은 없었어요.
          제가 준 건 함께 검토하자는 초대에 가까웠지 라이브 게시물을 바꾸라는 뜻은 아니었어요."

# friend06:beat:a:d-5:confession
수정 전: "공동 확인 원칙을 저도 깼다. 다솔의 위반을 탓하면서 같은 규칙을 제 손으로 무너뜨렸다."
수정 후: "공동 확인 원칙을 저도 깼어요. 다솔의 위반을 탓하면서 같은 규칙을 제 손으로 무너뜨렸어요."
```

**수정 범위**: friend-06의 partyA(윤해온) beatScript 전체 — deny/hedge/partial/blame/confession/evidence_hit 모든 대사에서 "~다" 종결을 "~요" 종결로 전환. partyB(최다솔)는 해요체를 사용하고 있으므로 수정 불필요.

**수정 방법**: 직접 수정 가능. GPT 재생성 불필요.
**수정 대상 파일**: `docs/ref/리뉴얼참고/gpt-batch/friend-06/friend-06-tells-beats.ts`
**예상 수정 건수**: partyA의 모든 beat (약 18개 대사)

## 4. lexicalHook 반영 기준

### 4-1. 반영률 기준

| 등급 | 반영률 | 판정 |
|------|--------|------|
| 양호 | Tell의 lexicalHook 중 50%+ 가 해당 party의 beatScript 어딘가에 등장 | PASS |
| 경고 | 25~49% | 수동 보강 가능 |
| 불량 | 0~24% | GPT 재생성 검토 |

**"등장"의 정의**: hook 문구가 beat의 `line` 필드에 정확히 또는 자연스러운 변형으로 포함됨.
예) hook "그때는 정말 급해서" → beat에 "그때는 정말 급해서" 또는 "그때 정말 급했거든요" 포함 → 인정

### 4-2. PASS 사건의 반영 사례 (neighbor-07)

```
Tell: neighbor07:a:tell:apology_preface
  hooks: ["죄송한데", "그때는 정말 급해서", "제가 오해했을 수도 있는데"]

Beat에서의 반영:
  d-1:deny "죄송한데 그때는 열쇠가 맞는지만..." → "죄송한데" ✅
  d-1:hedge "전 세입자도 그렇게 말해서..." → Tell의 borrowed_authority hook 반영
  → 반영률: 양호
```

### 5-1. family-02 callTerms 수정

**문제**: family-02의 관계는 시어머니(정미옥, 63) vs 아들(강선우, 35)이다.
그런데 callTerms가 며느리 시점으로 작성되어 있다.

```json
// 현재 (오류)
"partyA": {  // 정미옥 (어머니)
  "callTerms": {
    "toPartner": "선우야",        // ✅ 아들에게 → 맞음
    "toJudge": "제 며느리",       // ❌ 아들을 가리켜야 하는데 며느리로 표현
    "angry": "선우!"              // ✅ 맞음
  }
},
"partyB": {  // 강선우 (아들)
  "callTerms": {
    "toPartner": "어머님",        // ✅ 어머니에게 → 맞음
    "toJudge": "시어머님",        // ❌ 어머니를 가리키면서 "시어머님"은 며느리 시점 표현
    "angry": "어머님!"            // ✅ 맞음
  }
}
```

**수정값:**
```json
// 수정 후
"partyA": {  // 정미옥 (어머니)
  "callTerms": {
    "toPartner": "선우야",        // 유지
    "toJudge": "제 아들",         // "제 며느리" → "제 아들"
    "angry": "선우!"              // 유지
  }
},
"partyB": {  // 강선우 (아들)
  "callTerms": {
    "toPartner": "어머님",        // 유지
    "toJudge": "저희 어머니",     // "시어머님" → "저희 어머니"
    "angry": "어머님!"            // 유지
  }
}
```

**수정 대상 파일**: `src/data/cases/generated/family-02.json`

**참고**: `meta.familyRelation: "inlaw_mother_daughter_in_law"`도 사건 내용상 정확하지 않을 수 있으나, 사건의 핵심 갈등이 시어머니-며느리 축을 포함하므로(아들은 중간자) 유지해도 무방하다. 단, duo의 실제 양 당사자가 어머니-아들이므로 callTerms는 반드시 이 관계에 맞춰야 한다.
