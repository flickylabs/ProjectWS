## 요청 D: B-3 끼어들기 강화 엔진 설계

### 현재 구현 (ref-06, ref-08 참조)

현재 `gameEventTriggerEngine.ts`의 `checkInterjection`:
- 상대방 감정 40 이상 + 현재 쟁점 S1+ 상태일 때 발동
- 사실추궁/증거제시 시 더 잘 발생
- 쿨다운 3턴
- 감정 높으면 major

현재 `useActionDispatch`의 끼어들기 (ref-08 521~574줄):
- ① 강제: S4/S5 전이 시 100%
- ② 강제: S3 전이 (책임 전가) 시
- ③ 확률: 감정 기반 (phase3: 25%, angry면 +25%)
- 허용/제지 선택 → 지금은 자동 허용 후 resolveAndApply() 호출

### 요청하는 설계

**1. 패턴형 보장 규칙**
- 같은 대상 2턴 연속 + 상대 감정 40 이상 → 35~45%
- 같은 대상 3턴 연속 → **100% 보장**
- 쿨다운 3턴, 사건당 파티별 최대 2회

**2. 지식 사분면별 정보 차등**
dispute의 `quadrant`에 따라 끼어들기 내용이 달라져야 합니다:
- `a_only` / `b_only`: 상대는 감정적으로만 끼어들기 (정보 없음)
- `both_know`: 구체 디테일 반박 가능 (atom 기반 정보 노출)
- `shared_misconception`: 잘못된 확신을 더 강하게 꼬이게 만듦

**3. 허용/제지 리밸런스**
현재 제지가 항상 이득이라 선택이 안 됨. 리밸런스:
- **허용 시**: link reveal / trap signal / 새 atom 해금 / readiness 소폭 증가 / 흐름 리셋 (현재 쟁점 피로도 초기화)
- **제지 시**: 추궁 흐름 유지, authority -2, resentment flag (이후 해당 NPC의 순응 확률 -10%)

**4. Interjection beat 데이터 스키마**
1차에서 만든 `beats-v2-full`의 interjection beat 4개를 기준으로, 엔진이 어떤 beat를 어떻게 선택하는지:
- `quadrant` → 정보 수준 결정
- `beliefMode` → red_herring에서 오해 강화/약화
- `allowEffect` / `blockEffect` 필드

**5. 기존 코드 통합 포인트**
- `gameEventTriggerEngine.checkInterjection()` 교체 or 확장
- `useActionDispatch` 521~574줄 교체
- `GameEventModal` 컴포넌트에 허용/제지 선택지 연결

### 출력
`interjection-v2-design.ts` — 타입 + 엔진 로직 + useActionDispatch 통합 스니펫

## 요청 E: B-1 NPC 확률 반응 — blueprintEngine 통합 설계

### 현재 구현 (ref-07 참조)

`blueprintEngine.generateBlueprint()`가 매 턴 stance/defenseMode를 **결정론적으로** 결정:
- `baseStance(lieState, trust)` → `soften/harden` 보정 → 최종 stance
- 같은 입력이면 항상 같은 결과

### 요청하는 설계

**1. 행동 품질 기반 가중 확률**
stance 결정 **이후**, beatSelector **이전**에 확률 레이어를 추가:

```
좋은 선택 (정답 질문유형 + 맞는 angleTag): 순응 75 / 저항 20 / 역공 5
보통 선택: 순응 55 / 저항 30 / 역공 15
나쁜 선택 (affinityGrade worst): 순응 35 / 저항 40 / 역공 25
```

**역공 상한: 10~15%**

**2. archetype 보정**
- `avoidant`: 순응 +10, 역공 거의 0
- `confrontational`: 순응 -15, 역공 +10
- `cold_logic`: 순응 -10, 저항 +15
- 나머지 archetype별 보정도 제안해주세요

**3. 파이프라인 위치**
```
blueprintEngine.generateBlueprint() → stance 결정
→ [여기] npcReactionResolver → 순응/저항/역공 판정
→ beatSelectorV2 → beat 선택 (stance를 저항/역공으로 override 가능)
```

**4. "정답 행동이 무효가 되지 않게" 규칙**
- 순응: 기존 효과 100%
- 저항: 효과 50~60%, stance 하나 하향 (partial → hedge 등)
- 역공: 효과 0~30%, stance 하향 + 반격 beat 선택 + 권위 -1

**5. 캐릭터 패턴 vs 사건 패턴 분리**
"avoidant는 공감에 약하다"는 읽히되, "이 사건에서 공감 3번 하면 이긴다"는 읽히면 안 됨.
이를 위해 `archetype × lieMotive × disputeRole × linkState` 조합이 확률에 어떻게 반영되는지.

### 출력
`npc-reaction-v2-design.ts` — 타입 + 확률 계산 함수 + blueprintEngine/beatSelector 통합 스니펫

## 요청 F: 미터 단계화 + 사건카드 조건 해금

### F-1. 미터 수치 → 단계형 표시

**현재** (ref-09 StateTransitionFeedback.tsx의 QuestionMeterHUD):
- 모순토큰: 5개 pip으로 정확히 표시
- 누설미터: 0~100% 바로 표시
- 신뢰창구: 0~100% 바로 표시

**변경**:
```
모순토큰 → 그대로 유지 (이산형이라 이미 충분히 추상적)
누설미터 → 4단계: 안정 / 흔들림 / 위험 / 폭발직전
신뢰창구 → 4단계: 닫힘 / 살짝 / 열림 / 완전개방
```

각 단계의 임계값과 UI 표현(아이콘/색상/텍스트)을 설계해주세요.
그리고 **하드코어 모드** (수치 그대로 표시)용 토글도 포함.

### F-2. 사건카드 해금 조건 변경

**현재**: 턴 4 이후 무조건 해금 (고정)

**변경**: 아래 조건 중 **하나라도** 만족 시 해금
1. 첫 균열 발생 (아무 쟁점에서 S0→S1 이상 전이)
2. 모순토큰 2개 이상 축적
3. 신뢰창구 40 이상
4. 턴 6 도달 (안전장치 — 조건이 하나도 안 맞아도 턴 6이면 해금)

### F-3. readiness 일부 숨기기

현재 readinessScore가 간접적으로 노출됨.
판결 가능 여부는 표시하되, **구체 점수와 세부 조건은 숨기는** 방식.
"판결 가능까지 남은 것"을 **텍스트 힌트**로만 표시:
- "아직 충분한 근거가 없습니다"
- "심문이 진행되고 있습니다"
- "판결을 내릴 수 있습니다"

### 출력
`meter-staging-v2-design.ts` — 단계화 함수 + 사건카드 해금 조건 함수 + readiness 힌트 함수 + UI 스펙

## 요청 G: Phase 6/Result AI 프롬프트 재설계

### 문제
Phase 3가 스크립트화되면, Phase 6(조정) / Result(후일담) AI에게 넘기는 맥락이 달라집니다.
현재는 `dialogueLog`(전체 대화 기록)을 LLM에 넘기는데, 스크립트 Phase 3의 대화 기록은 **LLM이 생성한 것이 아니라** 선택된 beat의 나열이라 AI가 맥락을 잡기 어려울 수 있습니다.

### 요청하는 설계

**1. Phase3StructuredLog 활용 프롬프트**
1차에서 정의한 `Phase3StructuredLog`를 받아서:
```ts
{
  revealedAtoms: string[]      // 드러난 사실들
  disprovedFakeIssues: string[] // 해소된 가짜 쟁점
  resolvedLinks: string[]       // 활성화된 쟁점 간 연결
  relationCoreRevealed: boolean // 관계 핵심 층 도달 여부
  playerStyleTags: string[]     // 플레이어 스타일 (압박형/공감형/증거형 등)
}
```

**2. Phase 6(조정) 프롬프트 템플릿**
- 양측이 인정한 사실 vs 아직 다투는 사실
- 가짜 쟁점이 해소되었는지 여부
- 플레이어 스타일에 따른 조정 톤 (압박형 → 직설적 조정안, 공감형 → 화해 중심)
- 관계 핵심 도달 여부에 따른 깊이

**3. Result(후일담) 프롬프트 템플릿**
- 판결 결과 + 양측 반응
- 해소된/미해소 쟁점 기반 서사
- 관계 전망 (관계 핵심 도달 여부에 따라 다름)

**4. Phase 3 스크립트 ↔ Phase 6 AI 톤 연결**
스크립트 Phase 3에서 나온 대사의 톤/수준을 Phase 6 AI가 이어받으려면 무엇이 필요한지.
예: "캐릭터별 마지막 감정 상태", "가장 강렬했던 순간의 beat ID", "NPC가 직전에 한 마지막 말"

### 참고
현재 Phase 6/Result AI 코드는 `llmDialogueResolver.ts` 안에 있지만, Phase 3 교체 후에는 분리될 예정입니다.

### 출력
`phase6-result-prompt-v2-design.md` — 프롬프트 템플릿 2종 + structuredLog 활용 가이드 + 톤 연결 전략
