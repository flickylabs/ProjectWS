7종 산출물(스키마 + selector + 피로도 + 배치 템플릿 + 샘플)을 검토했고, **전량 채택**합니다. 품질이 높습니다.

## 요청 A: spouse-01 structure-v2.json 생성

02-dispute-v2-schema.ts와 03-evidence-v2-schema.ts 스키마를 **spouse-01에 실제 적용**한 데이터를 만들어 주세요.

### A-1. Dispute 확장 (5개 쟁점 전부)

spouse-01의 쟁점 5개를 아래 기준으로 재분류합니다.

| 쟁점 | disputeKind | 근거 |
|------|-------------|------|
| d-1 (280만원 비밀 송금) | `core_truth` | 사건의 핵심 매듭 |
| d-2 (새벽 휴대폰 열람) | `core_truth` | 양측 신뢰 붕괴의 핵심 |
| d-3 (잘린 캡처 오해) | `red_herring` | 외도 의심이지만 실은 오해 |
| d-4 (150만원 우회 송금) | `sub_truth` | d-1의 하위 맥락 |
| d-5 (사전 상의 약속 위반) | `core_truth` | 양측 공통 쟁점 |

각 쟁점에:
- `depthLayers` 3층 (surface/motive/core) + 각 층의 `unlockCondition` + `lockedSummary` + `revealAtomIds`
- `uiStyle` (surface: card_expand, motive: card_expand, core: relation_core)

### A-2. d-3 misconception 프로파일

d-3은 `red_herring`이므로 `misconception` 필수:
- `beliefModeByParty`: a는 `knows` (진실을 알지만 숨김), b는 `misbelief` (진심으로 외도를 의심)
- `stages` M0~M4: 각 stage의 summary + npcMode
- `trapSignals`: 사건 데이터에서 실제 함정 단서 추출 (잘린 캡처, 모텔가 인근 위치 등)
- `truthExitEvidenceIds`: 어떤 증거로 오해가 해소되는지

### A-3. Link Edges (3~5개)

spouse-01의 실제 쟁점 간 관계를 반영한 링크:
- d-1 → d-5: `supports` (송금 사실이 밝혀지면 "먼저 깬 건 누구" 공격 가능)
- d-2 → d-3: `unlocks_layer` (휴대폰 열람을 파면 캡처 오해의 맥락이 열림)
- d-1 → d-4: `weakens_counter` (280만원 경위가 밝혀지면 150만원 방어가 약해짐)
- d-3 → d-1: `retaliation` (외도 의심을 밀면 d-1에서 반격 가능)
- (필요 시 1개 추가)

각 link에 `when` (minState/minLayer) + `effect` + `uiLabel`

### A-4. Evidence timing metadata (6개 증거 전부)

spouse-01의 증거 e-1~e-6 각각에:
```json
{
  "timing": {
    "intent": "expose|contextualize|corroborate|disarm_trap",
    "role": "establish|reframe|impeach|finish",
    "bestAtStates": [...],
    "weakAtStates": [...],
    "preferredQuestionTypes": [...],
    "preferredAngles": [...],
    "blockedVectorsHelp": [...],
    "criticalWindows": [...]
  }
}
```

참고: 증거 목록은 첨부한 `ref-01-spouse01-case.json`에 있습니다.

### A-5. Free question hooks

spouse-01에서 의미 있는 자유 질문 intent 6~8개:
- 각 hook에 `allowedAtStates`, `allowedIssueRoles`, `answerEnvelope`, `refusalTemplates` 2~3개

### A-6. Phase 3 log hints


## 요청 C: legacy 감쇠 우회 설계

### 문제
현재 `questionEffectEngine.ts`에는:
```ts
const DIMINISHING_FACTOR = 0.6
// consecutive >= 3이면 적용
const diminish = consecutive >= 3 ? DIMINISHING_FACTOR : 1
```
그리고 `applyMeterUpdate()`에서 `consecutiveSameType`을 추적.

새 `questionFatigueEngine`은 `angleTag` 단위로 더 세밀한 피로도를 관리.
**둘 다 적용하면 이중 패널티**.

### 요청
아래를 구체 코드 수준으로 설계해 주세요:

1. **`resolveQuestionEffect()`에 `externalMultiplier` 파라미터 추가**
   - 기본값 1.0 (하위 호환)
   - 새 피로도 엔진이 활성화되면 내부 `diminish`를 무시하고 `externalMultiplier`만 사용

2. **통합 파이프라인 코드 스니펫**
   ```
   handleQuestion에서:
   1. blueprintEngine.generateBlueprint(...)
   2. questionFatigueEngine.evaluateQuestionFatigue(...)
   3. resolveQuestionEffect(..., { externalMultiplier: assessment.finalMultiplier })
   4. beatSelectorV2.selectTurnPresentation(...)
   5. questionFatigueEngine.commitQuestionFatigue(...)
   ```

3. **기존 `consecutiveSameType` 필드를 제거하지 않되, V2 모드에서 무시하는 방법**
   - feature flag? 분기? 별도 함수?

4. **테스트 시나리오 3개**
   - "같은 angleTag 3회 연속 → 효과 35%"
   - "angleTag 전환 → 피로 리셋 → 효과 100%"
   - "같은 대상 5회 연속 (spotlight) → 45%로 감소"

### 참고 파일
첨부한 `ref-05-questionEffectEngine.ts`가 현재 코드 전문입니다.
