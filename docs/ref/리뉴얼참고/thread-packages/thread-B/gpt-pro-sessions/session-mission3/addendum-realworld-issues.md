# [추가자료] 7건 실전 테스트에서 발견된 캐릭터 톤 문제

---

## 1. archetype이 무력화되는 현상 (TC-2 실패)

### 문제
confrontational(대결형) NPC가 avoidant(회피형)처럼 말하는 현상:

```
오세린 (confrontational 설정):
기대: "그건 제가 확인한 거예요. 의심할 만했잖아요"
실제: "복잡한 상황과 많은 요소들이 얽혀 있어서 간단히 말하기 어려운 점이 있습니다"
```

### 원인
publicClaim이 archetype 무관하게 작성되어 있어서, LLM이 publicClaim 텍스트를 그대로 사용.
archetype 가이드가 무력해짐.

### 자유 질문 톤 설계에의 시사점
자유 질문 응답에서 archetype 톤을 **강제로 입히는** 후처리가 필요.
LLM에 맡기면 publicClaim 톤에 의해 무력화되므로, **템플릿 기반 톤 적용**이 더 안전.

## 2. 실제 NPC 응답의 톤 차이 (좋은 예시 vs 나쁜 예시)

### 좋은 차별화 (S5 단계)
```
한지석(avoidant) S5:
"네, 그 280만원은 공동계좌에서 9월 20일에 송금했습니다."
→ 담담하게 사실 인정, 부끄러움 표현

오세린(confrontational) S5:
"솔직히 말해서, 바보 같은 아내가 될까 두려웠습니다."
→ 감정적이지만 직접적, 자기반성
```

### 나쁜 차별화 (S0-S2 단계)
```
한지석(avoidant) S1: "그날 공동계좌에서 일정 금액이 나간 건 사실입니다.
                     그러나 이름 하나만 보고 다른 의미를 만들려는 건 시기상조입니다."

윤태성(cold_logic) S1: "그 이메일 체인과 회의록은 맞습니다. 다만, 그때 마감 압박
                      때문에 관련 표기는 뒤로 미룬 것입니다."
```
→ 둘 다 "맞습니다. 다만..." 패턴. avoidant와 cold_logic의 차이가 거의 없음.

## 3. 자유 질문 톤 설계 시 반영사항

**archetype 톤이 가장 강하게 드러나야 하는 위치:**
1. **opener** (문장 시작) — 여기서 캐릭터가 결정됨
2. **closer** (문장 마무리) — 여기서 여운이 결정됨
3. connector는 상대적으로 덜 중요

**금지 패턴 (클리셰):**
| 금지 | 출현 빈도 | 대체 |
|------|----------|------|
| "맞습니다. 다만~" | 15회+ | archetype별 다른 인정 패턴 |
| "~할 수밖에 없었습니다" | 7회+ | archetype별 다른 변명 패턴 |
| "복잡한 상황이~" | 5회+ | 구체적 1가지 상황 언급 |
| "여러 가지 요소가~" | 4회+ | 사용 금지 (보고서 톤) |
