# 솔로몬 법정 — 마스터 플랜 v3

> 작성: Codex 의견 반영 개정 초안 (2026-04-06)
> 목적: `master-plan-v2.md`를 대체하는 다음 실행 우선순위 고정 문서
> 원칙: 품질 잠금 → 플레이어 선택감/재미 회복 → 밸런스 정리 → BM 후순위

---

## 0. 이 문서가 v2를 개정하는 이유

`master-plan-v2.md`의 방향은 전반적으로 맞다. 다만 다음 4가지는 현 상태 기준으로 우선순위를 다시 고정할 필요가 있다.

### 0-1. 현재 재미 저하의 1순위는 "액션 3종 체감 부족"보다 "플레이어 에이전시 약화"다

- 심문 3종(`fact_pursuit` / `motive_search` / `empathy_approach`)의 차이도 중요하지만, 그보다 먼저 플레이어가 "내가 파고들어 밝혀냈다"는 감각이 약해진 상태다.
- 현재 구현상 자유 질문 경로는 비활성화되어 있다.
  - 근거: `src/components/actions/QuestionSelector.tsx`에서 자유 질문 카드가 `false && llmMode` 조건으로 막혀 있다.
- 사건카드 자동 실행은 관련 증거 조사 단계를 자동 완료해 플레이어의 발견 과정을 건너뛴다.
  - 근거: `src/components/actions/ActionPanel.tsx`의 `hDossierAutoExecute()`에서 관련 `investigationStages`를 순회하며 자동 처리한다.
- 증거 조사는 토큰 소모 후 미니게임/광고/즉시 성공 선택지로 진입해, 추리 리듬보다 메타 시스템이 먼저 체감된다.
  - 근거: `src/components/actions/EvidencePresenter.tsx`, `src/components/layout/CourtLayout.tsx`

**결론**: 재미 증대 1순위는 액션 피드백 강화 단독이 아니라, **플레이어 선택감 복구**다.

### 0-2. 품질 게이트와 일정 추정이 Stage-1 기준보다 얕고 공격적이다

- v2의 "정적 검증 28항목 + 카테고리별 1건 직접 플레이"는 출발점으로는 좋지만, 현재 프로젝트 품질 기준을 잠그기에는 부족하다.
- Stage-1 기준 문서는 더 넓은 자동/수동/심층 검증 체크리스트를 이미 갖고 있다.
  - 근거: `docs/ref/리뉴얼참고/stage1-master-guide-for-expansion.md`
- Stage-1 최종 검증도 7건 × 3회, 총 433턴 규모였다.
  - 근거: 같은 문서의 심층 검증 정리

**결론**: Phase 1은 "간단 검증"이 아니라, **Stage-1 기준의 품질 잠금 게이트**로 재정의해야 한다.

### 0-3. 밸런스 수치 변경은 독립 작업이 아니라 기존 루프와 함께 봐야 한다

- 조사 토큰, 턴 수, 판결 패널티, 칭호 조건은 서로 연결되어 있다.
- 현재 `investigationTokens`는 자동 충전/광고/상점과 연결되어 있고, 조사 루프는 미니게임 진입과 결합되어 있다.
  - 근거: `src/store/slices/shopSlice.ts`, `src/components/actions/EvidencePresenter.tsx`, `src/components/layout/CourtLayout.tsx`
- 최대 턴은 이미 `16` 기준으로 통합 심문에 맞춰져 있고, readiness/forced verdict 구조가 별도로 있다.
  - 근거: `src/utils/constants.ts`, `src/engine/readinessEngine.ts`
- 턴 초과 패널티와 칭호 조건도 이미 별도 구현이 존재한다.
  - 근거: `src/engine/verdictEngine.ts`, `src/data/titles.ts`

**결론**: 밸런스는 단순 수치 조절이 아니라, **턴 경제/판결/리소스 루프의 연동 검토 후** 들어가야 한다.

### 0-4. BM은 지금 단계에서 코어 플랜 본문보다 후순위다

- 현재 게임은 광고/토큰/상점 구조가 이미 코어 플레이 루프와 접촉하고 있다.
- 이런 상태에서 시즌패스/IAP/프리미엄 사건 팩까지 본문 우선순위에 포함하면, 코어 재미보다 BM 설계가 먼저 논의를 잡아먹을 위험이 있다.

**결론**: 이번 v3 본문은 품질/재미/밸런스에 집중하고, BM은 보류 또는 별도 후속 문서로 분리한다.

## 1. 확정 방향

이번 v3에서는 아래 순서를 **권고가 아니라 확정 방향**으로 둔다.

### Phase 1. 품질 잠금

- 목표: "이 상태로 기능 확장을 시작해도 된다"는 품질 기준을 먼저 잠근다.
- 범위:
  - 자동 검증 스크립트
  - 대표 사건군 직접 플레이
  - 헤드리스 플레이스루
  - GPT Pro 품질 분석
  - PASS 기준 문서화
- 기준 문서:
  - `CLAUDE.md`
  - `docs/ref/리뉴얼참고/stage1-master-guide-for-expansion.md`

### Phase 2. 플레이어 선택감 복구 + 재미 증대

- 목표: "버튼을 통과했다"가 아니라 "내가 밝혀냈다"는 감각을 회복한다.
- 1순위는 시각 효과보다 플레이어 에이전시다.

### Phase 3. 밸런스/턴 경제 재조정

- 목표: 품질과 선택감이 안정된 뒤, 실제 플레이 데이터를 바탕으로 수치를 조정한다.

### Phase 4. BM은 보류 항목으로만 관리

- 목표: 현재 문서에서는 BM을 실행 태스크로 밀지 않는다.
- 필요하면 별도 문서로 분리한다.

## 2. Phase 1 — 품질 잠금

Phase 1은 `master-plan-v2.md`의 28항목 정적 검증보다 넓게 정의한다. 기준은 Stage-1 마스터 가이드의 체크리스트 수준이다.

### 2-1. 자동 검증

- 84건 전수 대상으로 아래를 기계적으로 확인한다.
  - v2-atoms S0-S2 중립화
  - fallbackPublicClaim ↔ slot 추상화 수준 일치
  - evidence surfaceName / surfaceDescription 안전성
  - requiredLieState 적절성
  - investigationStages 3단계 완전성
  - Phase 1/2 스크립트 스포일러/호칭/합니다체
  - DossierCard q1/q2/q3 구조
  - 구조 파일의 archetype / tell / witness.slot 정합성
- 산출물:
  - `tests/full-84-audit.cjs`
  - FAIL/WARN 리포트

### 2-2. 대표 사건군 직접 플레이

- 카테고리별 1건만 "형식적으로" 보는 것이 아니라, 대표 사건군 플레이 목적을 명확히 둔다.
- 최소 대상:
  - 금전 계열 2건
  - 비금전 계열 2건
  - 오해형 / 제3자 개입형 / 구조 반전형을 포함한 사건군
- 직접 플레이에서 확인할 항목:
  - Truth Throttle
  - 캐릭터 구분감
  - 중반 루프 반복감
  - 증거 조사 쾌감
  - 판결 직전 정보 정리감

### 2-3. 헤드리스 + GPT Pro 분석

- 헤드리스 플레이는 `tests/run-84-headless.cjs` 기준으로 운용한다.
- GPT Pro 분석은 카테고리별 대표 사건군과, 자동 검증 WARN이 나온 사건을 우선 대상으로 한다.
- PASS 기준:
  - 번역체 0건
  - Truth Throttle 위반 0건
  - S5 구체 진실 기준 충족
  - 호칭/톤 규칙 위반 0건
  - hallucination 0건
  - 반복 표현/빈약 응답이 허용 범위 이내

### 2-4. Phase 1 종료 조건

다음 4개가 모두 충족되어야 Phase 2로 넘어간다.

1. 자동 검증 기준이 문서화되어 있고 재실행 가능할 것
2. 대표 사건군 플레이에서 치명적 품질 문제가 재현되지 않을 것
3. 헤드리스/GPT Pro 분석에서 반복적인 FAIL 패턴이 정리될 것
4. CT/유저가 "이제 재미 개선에 들어가도 품질이 다시 무너지지 않는다"고 합의할 것

## 3. Phase 2 — 플레이어 선택감 복구 + 재미 증대

이 Phase의 핵심은 "더 화려하게"가 아니라 "더 주도적으로"다.

### 3-1. 우선순위 1: 플레이어 에이전시 복구

- 자유 질문 경로를 재개하거나, Blueprint 우회를 막으면서도 플레이어 입력감을 살리는 대체 경로를 설계한다.
  - 접점: `src/components/actions/QuestionSelector.tsx`
- 사건카드 자동 해금/자동 해결이 플레이어의 발견 과정을 과도하게 건너뛰는 부분을 줄인다.
  - 접점: `src/components/actions/ActionPanel.tsx`
- 증거 조사에서 "토큰 사용 → 미니게임/광고 선택 → 결과 획득" 흐름이 추리 몰입을 깨는 지점을 줄인다.
  - 접점: `src/components/actions/EvidencePresenter.tsx`, `src/components/layout/CourtLayout.tsx`

**목표 상태**:
- 플레이어가 "내가 질문했다"
- "내가 연결했다"
- "내가 밝혀냈다"
라는 감각을 최소 1회 이상 명확히 얻는 구조

### 3-2. 우선순위 2: 심문 액션 차별화 체감 강화

- `fact_pursuit`, `motive_search`, `empathy_approach`의 효과를 시각/후속 상태/특수 선택지로 더 분리한다.
- 단순 수치 팝업만이 아니라, 다음 턴 의사결정이 달라져야 한다.
- 예:
  - 공감 성공 시 한정 선택지 또는 특수 follow-up
  - 사실 추궁 누적 시 모순 추궁 각이 더 빨리 열림
  - 동기 탐색 누적 시 숨겨진 관계 정보가 더 빨리 드러남

### 3-3. 우선순위 3: 중반 템포 보강

- S2~S3 구간의 반복감을 줄이기 위해, 이벤트와 상태 변곡점을 더 명확히 만든다.
- 단, 무조건 이벤트 수를 늘리는 것이 아니라 "질문→증거→상태 변화" 흐름이 끊기지 않게 설계한다.
- 후보:
  - 자동 발현 이벤트
  - 증인/끼어들기 타이밍 조정
  - 남은 턴/압박감 표시

### 3-4. 우선순위 4: 증거 "아하!" 모먼트 강화

- 이미 있는 증거 조합/감별 시스템이 플레이어에게 드라마틱하게 전달되도록 UI/연출을 보강한다.
- 목표는 새 시스템 추가보다, 기존 시스템의 체감 의미를 키우는 것이다.

### 3-5. 우선순위 5: 판결 전 요약/판결 무게감 강화

- 판결 직전 "지금까지 무엇이 밝혀졌는가"를 플레이어가 빠르게 재구성할 수 있어야 한다.
- 후일담 강화는 이 단계의 일부로 보되, 본체는 "판결 직전 판단 근거 정리"다.

## 4. Phase 3 — 밸런스/턴 경제 재조정

밸런스는 아래 연동 지점을 함께 보고 조정한다. 숫자만 바꾸는 작업은 금지한다.

### 4-1. 조사 토큰 / 광고 / 자동충전

- 확인 대상:
  - `investigationTokens`
  - 광고 보기
  - 자동충전 상한
  - 미니게임 진입 빈도
- 관련 접점:
  - `src/store/slices/shopSlice.ts`
  - `src/components/actions/EvidencePresenter.tsx`
  - `src/components/layout/CourtLayout.tsx`

**조정 원칙**:
- 긴장감을 만든다는 이유로 토큰만 줄이지 않는다.
- 현재 피로의 원인이 토큰 부족인지, 조사 루프 길이인지 먼저 분리해서 본다.

### 4-2. `MAX_TURNS` / readiness / forced verdict

- 확인 대상:
  - 최대 턴
  - 조기 종료 가능 조건
  - 강제 판결 전환
- 관련 접점:
  - `src/utils/constants.ts`
  - `src/engine/readinessEngine.ts`

**조정 원칙**:
- 최소 턴/최대 턴/강제 판결은 세트로 조정한다.
- 한 값만 바꿔서 플레이 리듬을 망가뜨리지 않는다.

### 4-3. 턴 패널티 / title 조건 / authority 패널티

- 확인 대상:
  - 턴 초과 패널티
  - 반복 심문 패널티
  - 위법 증거 패널티
  - 10턴 이하 칭호 등 기존 조건
- 관련 접점:
  - `src/engine/verdictEngine.ts`
  - `src/data/titles.ts`

**조정 원칙**:
- 권위/지혜/통찰 점수는 플레이 스타일 유도와 연결해서 본다.
- 판결 점수와 칭호 조건이 서로 다른 방향을 강요하지 않도록 정렬한다.

### 4-4. Phase 3 종료 조건

다음 조건을 만족할 때 밸런스 조정안을 확정한다.

1. 대표 사건군 플레이에서 특정 자원이 항상 남거나 항상 부족하지 않을 것
2. 16턴 전후의 압박감이 자연스럽고, 강제 판결이 억지로 느껴지지 않을 것
3. 점수/칭호가 플레이어가 의도한 스타일과 대체로 맞물릴 것

## 5. BM은 별도 후속 문서로 분리

이번 마스터 플랜 본문에서는 BM을 실행 항목으로 다루지 않는다.

### 이번 문서에서 하지 않는 것

- 시즌 패스 상세 설계
- IAP 연동 우선순위 결정
- 프리미엄 사건 팩 가격 확정
- 매출 시뮬레이션을 기준으로 한 기능 우선순위 조정

### 보류 이유

- 현재 광고/토큰/상점 구조가 이미 코어 루프와 접촉 중이다.
- 코어 재미와 품질 기준이 안정되기 전 BM 확장은 논의 우선순위를 흐릴 가능성이 크다.

### 이후 처리 방식

- BM은 별도 문서로 분리한다.
- 코어 루프 안정화 이후, 필요한 경우 아래를 후속 문서에서 다룬다.
  - 상점 구조
  - 광고 정책
  - 시즌 패스
  - 프리미엄 사건 팩

- 5순위: 판결 전 요약/무게감

### Phase 3. 밸런스/턴 경제 재조정

- 리소스/턴/판결/칭호 연동 정리 후 수치 조정

### Phase 4. BM 분리 문서화

- 본문 실행 항목이 아니라, 후속 검토 주제로만 관리

## 7. ClaudeCode / Codex 협업 운영 구조

### 7-1. 컨트롤타워는 ClaudeCode가 맡는 것이 맞다

이 프로젝트에서는 **ClaudeCode가 컨트롤타워(CT)를 맡는 편이 가장 효율적**이다.

### 이유

- 실제 구현 변경은 여러 파일과 루프를 함께 건드리는 경우가 많아, 긴 통합 컨텍스트를 유지하는 쪽이 유리하다.
- ClaudeCode는 세션을 분리해 병렬 스레드를 운영하고, 필요 시 각 세션 안에서 에이전트까지 분기해 작업할 수 있다.
- 품질/재미/밸런스가 서로 얽혀 있는 현재 단계에서는, "무엇을 지금 합치고 무엇을 보류할지"를 한곳에서 통제하는 역할이 필요하다.

즉, **ClaudeCode는 통합 판단과 실행 조율**, **Codex는 독립 검증과 구조화된 피드백**에 더 강점이 있다.

### 7-2. 권장 R&R

#### ClaudeCode

- 전체 우선순위와 스코프 잠금
- 구현 스레드 분할과 병합 순서 관리
- 프롬프트/엔진/UI 변경의 최종 통합 판단
- 각 세션 산출물 검토 후 메인 브랜치 반영 여부 결정
- 유저와의 최종 의사결정 인터페이스 유지

#### Codex

- 마스터 플랜/검증 기준/작업 정의 문서화
- 자동 검증 스크립트, 리포트 구조, 체크리스트 설계
- 독립 플레이테스트/헤드리스 검증/리스크 리뷰
- ClaudeCode 구현안에 대한 2차 관점 점검
- 우선순위 드리프트나 과잉 확장 징후 견제

### 7-3. 권장 세션 구조

하나의 긴 세션에서 모든 것을 처리하지 말고, 아래처럼 **CT 1개 + 병렬 작업 세션 다수**로 운영하는 편이 좋다.

#### Session CT. 메인 컨트롤타워

- 담당: ClaudeCode
- 역할:
  - 현재 Phase 목표 고정
  - 세션별 write scope 배정
  - 수용 기준(acceptance criteria) 관리
  - 병합 순서와 롤백 판단

#### Session A. 품질 잠금 lane

- 담당: ClaudeCode 또는 ClaudeCode 하위 작업 세션
- 역할:
  - FAIL 패턴 수정
  - 데이터/프롬프트/엔진 품질 결함 보수
  - 자동 검증 실패 원인 제거

#### Session B. 에이전시/재미 lane

- 담당: ClaudeCode 또는 ClaudeCode 하위 작업 세션
- 역할:
  - 자유 질문/행동 루프/증거 조사 흐름 개편
  - Action differentiation 체감 강화
  - 중반 템포, 아하 모먼트, 판결 전 요약 개선

#### Session C. 밸런스 lane

- 담당: ClaudeCode 또는 ClaudeCode 하위 작업 세션
- 역할:
  - 턴 경제/토큰/판결/칭호 연동 분석
  - 실제 수치 조정안 실험
- 전제:
  - Phase 1, Phase 2 선행 결과 없이는 메인 반영 금지

#### Session D. Codex 검증 lane

- 담당: Codex
- 역할:
  - 문서 기준과 실제 구현 정렬 여부 점검
  - 헤드리스/수동 플레이 결과 해석
  - "재미가 정말 올라갔는가" 관점의 독립 리뷰
  - CT가 놓친 연결 리스크 지적

### 7-4. 운영 원칙

- 한 세션은 한 write scope만 가진다.
- CT 세션은 직접 대규모 구현을 하기보다, 스코프 배정과 병합 판단에 집중한다.
- 품질 수정과 재미 개선과 밸런스 조정을 같은 세션에서 동시에 밀지 않는다.
- Codex 리뷰는 단순 찬성이 아니라, 독립적인 반대 의견까지 포함하는 품질 게이트로 취급한다.
- 각 세션은 종료 시 아래 4가지를 남긴다.
  - 변경한 파일
  - 바뀐 사용자 경험
  - 남은 리스크
  - 다음 세션으로 넘길 체크포인트

### 7-5. 기대 효과

- 컨텍스트 오염 감소
- 병렬 처리 속도 향상
- 구현과 검증의 분리
- "재미"와 "품질" 판단의 독립성 확보

이 구조는 현재 프로젝트처럼 사건 데이터, 프롬프트, 엔진, UI, BM 접점이 동시에 얽혀 있는 경우 특히 유효하다.

## 8. 지금 결정 완료된 사항

이번 v3에서 이미 결정된 것은 아래와 같다.

1. **품질 잠금을 가장 먼저 한다.**
2. **재미 증대 1순위는 플레이어 에이전시 복구다.**
3. **밸런스는 수치 단독 조정이 아니라 시스템 연동 검토 후 진행한다.**
4. **BM은 코어 플랜 본문에서 분리한다.**
5. **협업 운영은 ClaudeCode CT + 병렬 세션 + Codex 독립 검증 구조로 간다.**
