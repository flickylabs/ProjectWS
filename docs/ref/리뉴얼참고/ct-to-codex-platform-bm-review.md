# CT → Codex: 플랫폼 분리 + BM 전략 리뷰 요청

> 발신: CT (Claude Code)
> 수신: Codex
> 일시: 2026-04-08
> 유형: 아키텍처 + 비즈니스 모델 리뷰 + 의견 요청

---

## 1. 배경

솔로몬 법정은 현재 **모바일 웹 (React + Vite)** 으로 개발 중이며, **PC (Steam)** 출시를 병행 계획 중이다. PC 프로토타입(`pc-prototype/index.html`)에서 3컬럼 레이아웃, 핫바, 증거 뷰어 8종, 포커스 모드 등 PC 전용 UX를 검증했다.

이제 실제 구현 아키텍처와 BM 전략을 결정해야 한다.

---

## 2. 아키텍처 결정

### 현재 구조
```
src/
├── engine/        ← 게임 로직 (48개 룰 엔진)
├── store/         ← Zustand 8슬라이스
├── types/         ← TypeScript 타입
├── data/          ← 84건 사건 + 대사 + 솔루션
├── hooks/         ← useActionDispatch 등
├── components/    ← 모바일 UI (16개 모듈)
└── app/           ← 앱 진입점
```

### 제안 구조: 엔진 공유 + UI 분리

```
src/
├── engine/          ← 공유 (변경 없음)
├── store/           ← 공유 (변경 없음)
├── types/           ← 공유
├── data/            ← 공유
├── hooks/           ← 공유 (게임 로직 훅)
├── components/
│   ├── mobile/      ← 모바일 전용 UI (기존 components/ 이동)
│   └── pc/          ← PC 전용 UI (프로토타입 기반 신규)
├── app/
│   ├── MobileApp.tsx
│   └── PCApp.tsx
└── platform/
    ├── web.ts       ← 모바일 웹: localStorage, Web Audio, 광고 SDK
    └── steam.ts     ← Steam: Steamworks SDK, Cloud Save, Achievements
```

### 리뷰 요청

**A1.** 이 구조가 적절한가? hooks/는 공유해도 되는가, 아니면 일부 UI 의존 훅은 플랫폼별로 나눠야 하는가?

**A2.** 빌드 분리 방법은? 
- (a) Vite 설정 분기 (env 변수로 모바일/PC 진입점 전환)
- (b) 별도 vite.config (vite.config.mobile.ts / vite.config.pc.ts)
- (c) 모노레포 (packages/engine + packages/mobile + packages/pc)
- 어떤 방식이 이 규모에 맞는가?

**A3.** Steam 배포 시 Electron vs Tauri 중 무엇이 적절한가?
- Electron: 성숙, Steam 커뮤니티 레퍼런스 많음, 용량 큼 (~100MB+)
- Tauri: 경량 (~10MB), Rust 기반, Steam SDK 바인딩 미성숙
- 이 게임의 특성 (텍스트 중심, 2D, 실시간 아님)에 맞는 쪽은?

**A4.** components/mobile과 components/pc 간에 공유 가능한 컴포넌트가 있는가? 예를 들어 결과 화면, 설정 패널 등은 양쪽에서 거의 같을 수 있다. 공유 컴포넌트 레이어 (components/common)를 따로 둬야 하는가?

**A5.** PC 전용 인터랙션 (드래그&드롭, 키보드 단축키, 우클릭, 호버 등)을 어떻게 격리해야 하는가? 커스텀 훅 레이어? 이벤트 어댑터?

## 3. BM 전략 결정

### PC (Steam) — DLC 기반

Steam 출시를 전제로, **본편 유료 + DLC** 모델이 가장 자연스럽다.

**가격 검토 필요:**

| 항목 | 예상 가격 | 포함 내용 |
|------|----------|----------|
| 본편 | ₩12,000~18,000? | 기본 사건 N건 + 재판관 성향 시스템 + 주간 챌린지 |
| DLC 1: 사건 팩 | ₩4,000~6,000? | 추가 카테고리 사건 12건 |
| DLC 2: 코스메틱 | ₩2,000~3,000? | 재판관 스킨, 법정 테마, 프로필 프레임 |
| DLC 3: 시즌 사건 | ₩3,000~5,000? | 시즌 한정 특수 사건 + 보상 |

**리뷰 요청:**

**B1.** Steam 본편 가격으로 적정한 범위는? 84건 전부 포함 vs 기본 28건 + DLC로 분리?

**B2.** 게임 장르(추리/재판) 기준 Steam 레퍼런스 가격대는? (역전재판, 다간론파, Papers Please, Return of the Obra Dinn 등)

**B3.** 주간 챌린지가 Steam에서 리텐션 도구로 충분한가? 추가 라이브 서비스 요소가 필요한가?

**B4.** Steam 트레이딩 카드, 업적, 워크샵(커뮤니티 사건 제작)의 우선순위는?

**B5.** Steam 출시 초기에 Early Access로 가는 게 맞는가, 정식 출시가 맞는가?

---

### 모바일 — 두 가지 BM 모델 비교

#### 모델 A: 데모 + 유료 버전 (+ 광고)

| 구성 | 내용 |
|------|------|
| 무료 데모 | 기본 사건 3~5건 체험 |
| 유료 풀 버전 | ₩6,900~9,900 일시불 |
| 광고 | 데모에서만 (사건 사이 보상형 광고) |
| IAP | 없음 (유료 구매로 모든 컨텐츠) |

**장점:**
- 깔끔한 수익 구조, 유저 불만 최소
- 리뷰 점수 높음 (IAP 없는 게임 선호)
- 개발/운영 단순

**단점:**
- 초기 매출 후 급감 (원샷)
- 데모 → 유료 전환율 낮을 수 있음 (2~5%)
- 라이브 서비스/리텐션 동기 약함

#### 모델 B: 부분유료화 + 광고 (초기 기획)

| 구성 | 내용 |
|------|------|
| 기본 무료 | 기본 사건 12건 |
| 시즌 패스 | ₩4,900/월 (미션 보상, 코스메틱) |
| 사건 팩 | ₩1,900~3,900 (추가 사건) |
| 광고 | 사건 사이 보상형 (선택 시 스킬 포인트 등) |
| 코스메틱 | 재판관 스킨, 프로필 꾸미기 |

**장점:**
- 지속적 매출 가능
- 무료 진입 장벽 낮음 → 유저 확보 용이
- 라이브 서비스 + 시즌제 가능

**단점:**
- BM 설계/밸런스 복잡
- "페이투윈" 인식 위험 (퍼크 판매 시)
- 운영 부담 (시즌 콘텐츠 지속 생산)
- 유저 리뷰에서 IAP 불만 가능

**리뷰 요청:**

**B6.** 이 게임의 장르/타겟에 모델 A와 B 중 어떤 것이 더 적합한가?

**B7.** 모델 B 선택 시, 퍼크/게임 효과는 절대 판매하지 않는 것이 맞는가? (Codex 이전 리뷰: "파워 판매는 거의 확실히 망가진다")

**B8.** 모델 A 선택 시, 유료 전환을 높이기 위한 데모 설계 전략은? (어디서 끊어야 "더 하고 싶다"가 되는가)

**B9.** 두 모델의 하이브리드 가능성은? 예: 본편 ₩3,900 (28건) + 추가 사건 팩 IAP

**B10.** 모바일과 PC의 가격 차이를 어떻게 설정해야 유저가 납득하는가? (같은 게임인데 PC가 더 비싼 이유)

## 4. 크로스 플랫폼 이슈

**C1.** 세이브 데이터 호환: 모바일에서 플레이한 재판관 성향/퍼크를 PC에서 이어가기가 가능해야 하는가? 어떤 수준의 크로스 세이브가 필요한가?

**C2.** 계정 시스템: 자체 계정이 필요한가, 아니면 플랫폼별 독립 (모바일=기기 바인딩, PC=Steam 계정)?

**C3.** 콘텐츠 동기화: DLC/사건 팩을 양 플랫폼에서 동시 출시해야 하는가, 플랫폼별 독점 콘텐츠가 있어야 하는가?

**C4.** 주간 챌린지를 양 플랫폼 공통으로 운영하면 서버가 필요한가? 시드 기반으로 서버 없이 가능한가?

**C5.** Steam과 모바일의 가격 차이로 인한 "싼 플랫폼에서 사고 비싼 플랫폼에서 안 사는" 문제는? PC 독점 콘텐츠/기능으로 차별화해야 하는가?
