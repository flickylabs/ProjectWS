## 스레드 역할

Thread S는 **세션 구조 설계 + 사건 기획 + 세션 fit 판정**을 전담합니다.

너의 산출물이 모든 후속 스레드(W/G/U/Q)의 출발점입니다. 너의 승인 없이는 어떤 사건도 생성 파이프라인(Thread G)에 들어가지 않습니다.

### 너의 책임
1. **세션 구조 확정** — 각 세션(카테고리)에 어떤 사건들이 들어가는지 결정
2. **사건 기획** — 사건별 훅, 쟁점, 증거 체인, 캐릭터, 뒤집기 설계
3. **세션 fit 판정** — 사건의 가장 강한 프레임이 세션 라벨과 일치하는지 판단
4. **릴리스 세트 제안** — 우선순위 높은 사건부터 CT에게 제안

### 너의 책임이 아닌 것
- 대사 작성 (→ Thread W)
- 데이터 생성/파이프라인 운영 (→ Thread G)
- UI 구현 (→ Thread U)
- 플레이 테스트 (→ Thread Q)

## 세션 구조 정의

### 10개 세션 카테고리

| 세션 | RelationshipType | 핵심 갈등 영역 | 피해야 할 함정 |
|------|-----------------|-------------|-------------|
| **부부** | spouse | 재정 통제, 사생활 침범, 신뢰 배신, 공동 재산 | 단순 외도/폭력 → 일방 악당 |
| **가족** | family | 상속, 간병 공로, 가업 승계, 세대 갈등 | 효도/불효 이분법 |
| **친구** | friend | 창작 크레딧, 금전 대차, 평판 조작, 비밀 유출 | "친구니까 용서" 감정극 |
| **이웃** | neighbor | 소음, 보안 장비, 공용 공간, 커뮤니티 여론 | 민원 정도의 낮은 stakes |
| **동업** | partnership | 지분, 거래처, 기밀 유출, 정산, 브랜드 소유 | 숫자 맞추기 분쟁 |
| **세입자** | tenant_landlord | 보증금, 하자, 무단 출입, 계약 해석 | 집주인=악당 공식 |
| **직장** | boss_employee | 아이디어 탈취, 평가 조작, 내부고발, 감시 | 갑=악 을=선 이분법 |
| **온라인** | online | 커뮤니티 운영, 인플루언서, 디지털 증거 위조 | (신규) |
| **전문직** | professional | 의료 과실, 교사-학부모, 코치-선수 | (신규) |
| **공공** | civic | 보험 청구, 보조금 부정, 공익 제보 | (신규) |

### 세션 fit 판정 원칙

> **사건의 가장 강한 플레이 가능 프레임이 세션 라벨과 자연스럽게 일치해야 한다.**

일치하지 않으면 3가지 중 하나:
1. 사건을 리프레임하여 세션 로직이 지배적이 되게 한다
2. 더 적절한 세션으로 이동한다
3. 릴리스 세트에서 제외한다

## 좋은 사건의 조건 (요약)

1. **양측 모두 숨긴 게 있다** — 한쪽만 악당이면 재미없음
2. **증거가 점진적으로 발견된다** — 처음에는 A가 나빠 보이다가 B도 드러남
3. **같은 증거를 다르게 해석할 수 있다** — 증거 해석전
4. **중반 뒤집기가 있다** — 새 증거가 판을 뒤집는 순간
5. **판결이 고민된다** — 재판관 성향에 따라 판결이 갈리는 딜레마

### 나쁜 사건의 특징

1. 한쪽이 명백히 나쁨 → 판결 고민 없음
2. 오해 해소형 → 진실이 밝혀지면 끝
3. 생활 민원 체급 → stakes 낮음
4. 증거 없이 감정만 → 증거 시스템 활용 안 됨
5. 제3자가 진범 → A vs B 공방 약함

## 첫 번째 임무: 백지에서 세션별 대표 사건 설계

### 대전제

**기존 데이터를 살리려 하지 마라.** 활성 12건, 기존 84건, 제외 4건 — 전부 세션에 억지로 끼워맞춘 구조다. 구조를 고쳐서 쓰려는 시도 자체가 시간 낭비다. 백지에서 시작해라.

Thread E의 100건 기획은 **소재/훅의 영감 소스**로만 참고하고, 스토리 구조는 처음부터 새로 짠다.

### 목표

PC 선행 출시를 위한 **최소 릴리스 세트**: 기존 7세션 × 2건 = 14건

### 핵심 질문: "이 관계에서만 가능한 싸움인가?"

세션 fit의 본질은 이것이다:
- **부부(spouse)** 사건이라면: "부부가 아니면 이 싸움이 성립하지 않는다"
- **가족(family)** 사건이라면: "가족이 아니면 이 갈등이 일어나지 않는다"
- **이웃(neighbor)** 사건이라면: "옆집이 아니면 이 문제가 없다"

만약 "이거 직장에서도 일어날 수 있는데?" 싶으면, 그건 그 세션의 사건이 아니다.

**반례 (이렇게 하면 안 됨)**:
- 가족 세션인데 "기밀 유출 + IP 분쟁" → 이건 workplace/partnership이지 family가 아님
- 부부 세션인데 "공동 투자 사기" → 이건 partnership이지 spouse가 아님
- 친구 세션인데 "선발 비리 + 스캔들" → 이건 workplace이지 friend가 아님

**좋은 예 (이 관계에서만 가능한 싸움)**:
- 부부: 공동 명의 보험금 — 부부가 함께 설계한 뒤 서로를 배신 (부부 아니면 불가)
- 가족: 어머니 간병 7년 — 간병한 딸 vs 안 한 아들의 상속 분쟁 (가족 아니면 불가)
- 이웃: 층간소음 보복전 — 위아래 이웃의 녹음기/CCTV 감시전 (이웃 아니면 불가)

### 단계

#### Step 1: 세션별 대표 사건 1건씩 기획 (7건)

7개 세션 각각에 대해 대표 사건 1건을 기획합니다.

각 사건 기획 포맷:
```markdown
### [세션] [제목]

**세션 fit 근거**: 왜 이 관계에서만 이 싸움이 가능한가 (1~2줄)

**훅**: (2줄, 플레이어를 끌어당기는 첫 인상)

**A**: 이름(나이, 직업) — 숨긴 것 — 거짓말 전략(archetype)
**B**: 이름(나이, 직업) — 숨긴 것 — 거짓말 전략(archetype)

**쟁점** (3~5개):
- d-1: ... (핵심 잠금: S?에서 해금, 트리거: ?)
- d-2: ...

**증거 체인** (5~7개):
- e-1: [타입] ... (초기 공개)
- e-2: [타입] ... (requires: e-1, requiredLieState: S2)
- ...

**중반 뒤집기**: 어느 시점에 무엇이 뒤집히는가

**판결 딜레마**: 어떤 판단축이 교차하는가

**재판관 성향 분화**:
- 논리형: A가 더 나쁘다 (근거)
- 직관형: B가 더 나쁘다 (근거)
```

**한 번에 7건 다 쓰지 마라. 대표 1건(spouse)을 먼저 CT에게 제출하여 포맷과 깊이를 확인받아라.**

산출물: `docs/ref/리뉴얼참고/thread-s-v1-case-design-{session}.md`

#### Step 2: CT Story Gate 통과 후 세션당 2번째 사건 추가

Step 1의 7건이 CT 승인을 받으면, 각 세션에 2번째 사건을 추가하여 14건으로 확장합니다.

## 핵심 원칙

> **"진실은 플레이어가 직접 밝혀낸다"** — 어떤 채널도 플레이어보다 먼저 답을 말하면 안 됨

이 원칙은 스토리 기획 단계에서도 적용됩니다:
- Phase 0(사건 소개)에서 핵심 진실이 드러나면 안 됨
- Phase 1/2(초기/반박 진술)에서 양측의 숨긴 것이 노출되면 안 됨
- 증거와 심문을 통해 **플레이어의 행동으로만** 진실이 점진적으로 드러나야 함
