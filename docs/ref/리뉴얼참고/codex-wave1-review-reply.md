## Codex 회신

### Q1 (실행 순서): [동의, H1을 먼저 처리해 에이전시 파손을 줄이고, Idea 2는 `stateTransitionHelper.ts`/`StateTransitionFeedback.tsx` 중심이라 독립성이 높습니다. Idea 3과 Idea 1은 둘 다 `QuestionSelector.tsx`를 건드리므로 마지막에 한 번에 합치는 편이 맞습니다. 실무적으로는 `H1 → 2 → (3+1 단일 QuestionSelector 패스)`로 이해하면 됩니다.]

### Q2 (효과성 규칙): [불일치, 현재 엔진 기준으로 `fact_pursuit`는 `deny | hedge | blame`일 때만 유효하고 `partial`에서는 강효과가 아닙니다 (`src/engine/questionEffectEngine.ts:214-218`, `src/hooks/useActionDispatch.ts:669`, `src/hooks/useActionDispatch.ts:1087`). 따라서 `S0~S2 strong`은 과장이고, `S0~S1 strong / S2 normal / S3 strong` 쪽이 더 맞습니다. `empathy_approach`도 일반 `S3+ strong`은 넓습니다. 실제 lieStateMachine의 신뢰 기반 점프는 `S3→S5`가 `third_party_protection + retaliationWorry < 30`일 때만 열리고, 범용적으로 강한 구간은 `S4→S5`입니다 (`src/engine/lieStateMachine.ts:88-106`). 따라서 `S4 strong / S3 conditional / 그 외 normal`이 안전합니다. `cold_logic && contradictionTokens < 2`를 일괄 weak로 두는 것도 단순화가 과합니다. `contradictionTokens`는 party-level 미터일 뿐 증거 보유의 직접 대리값이 아니고, `S4`에서는 토큰이 낮아도 공감이 먹힐 수 있습니다. 최소한 `S0~S2 && contradictionTokens < 2`로 좁혀야 합니다.]

### Q3 (angleTag 단순화): [불충분, 현재 피로도 key는 `party:disputeId:angleTag`이고 (`src/engine/questionFatigueEngine.ts:19-24`, `src/engine/questionFatigueEngine.ts:242-243`), 실제 `angleTag`는 같은 `questionType` 안에서도 갈립니다. 예를 들어 `fact_pursuit`는 `timeline` 또는 `responsibility`, `motive_search`는 `motive` 또는 `emotion`으로 나뉩니다 (`src/engine/beatSelectorV2.ts:188-195`). `questionType` 기준 최대 streak만 읽으면 현재 선택이 실제로는 fresh인데도 경고가 뜨는 over-warning이 생깁니다. UI를 아주 약한 주의 문구로만 쓸 거면 타협 가능하지만, `효과 낮음`이나 `교착` 판단까지 붙일 용도로는 정확도가 부족합니다. beatSelector 전체 의존 대신, `QuestionSelector` 안에 `deriveAngleTag`의 축약판을 두거나 최소 `questionType + layerBand` 수준까지는 맞추는 편이 낫습니다.]

### Q4 (이중 패널티): [해소됨, 현재 V2 질문 경로는 이미 `externalMultiplier`와 `bypassLegacyDiminish: true`를 같이 넘기고 있어서 (`src/hooks/useActionDispatch.ts:743-746`), `questionEffectEngine`의 legacy 0.6 감쇠와 `questionFatigueEngine`의 multiplier가 동시에 먹는 문제는 본 경로에서는 막혀 있습니다. 따라서 교착 UI를 숫자 패널티가 아닌 안내로만 유지하면 3중 패널티는 아닙니다. 권장 처리는 두 가지입니다. 첫째, fatigue를 쓰는 모든 `applyQuestionEffect` 호출에서 `bypassLegacyDiminish`를 강제합니다. 둘째, UI 경고는 선택 차단이나 추가 수치 감쇠 없이 정보 전달만 하도록 고정합니다. 다만 V2 미사용 fallback 호출(`src/hooks/useActionDispatch.ts:1088`)은 아직 bypass가 없으므로, 적용 범위를 명시적으로 정리해야 재발을 막을 수 있습니다.]

### Q5 (추가 리스크): [있음, 가장 큰 리스크는 UI가 실제보다 많은 것을 약속하는 경우입니다. `questionEffectEngine`가 내보내는 `suppression_leak`, `defense_weaken`, `confession_window`, `lie_transition_bonus`는 현재 store에서 실효 처리되지 않고 사실상 dormant입니다 (`src/store/useGameStore.ts:323-335`, `src/engine/questionEffectEngine.ts:31-37`). Wave 1 UI가 이 효과를 “이미 적용된 실제 변화”처럼 보이게 만들면 플레이 체감이 깨집니다. 두 번째는 H1 pacing 리스크입니다. `ActionPanel.tsx`의 dossier 자동 조사 완료를 빼면 (`src/components/actions/ActionPanel.tsx:349-357`) 즉시 이어지는 추천 증거/다음 행동 유도가 약하면 카드가 nerf처럼 느껴질 수 있습니다. 세 번째는 검증 리스크입니다. 이번 Wave 1은 `QuestionSelector`, `questionFatigueEngine`, `questionEffectEngine`, `stateTransitionHelper`, `StateTransitionFeedback`가 교차하므로 S0~S5, target switch, dossier_execute reset, V2 fallback 분기까지 묶은 회귀 체크가 필요합니다. 네 번째는 Phase 3 충돌 리스크입니다. Wave 1에서 `효과 낮음` 임계값을 박아 넣고 나중에 턴 경제/밸런스 수치를 바꾸면 UI 문구가 바로 낡습니다. threshold는 중앙화해 두는 편이 안전합니다.]

### 위임 A/B: [확인, A는 누락분 포함 완료됐습니다. `tenant-01~12`의 `r1~r3` 36개와 `partnership-11~12`의 `r1~r3` 6개를 포함해 확인했습니다. 전체 partnership도 `r1~r3` 기준 36개가 존재합니다. B 산출물은 `tests/84-llm-quality-report.md`에 정리돼 있습니다. 단, 해당 리포트에도 명시했듯 이번 전수 동적 검증은 NPC 대사 중심이고, 증인/끼어들기/자유 질문/후일담은 전용 harness 부재로 정적 또는 부분 검토입니다.]

### 추가 의견: [Wave 1에서 바로 구현 가치가 큰 것은 “효과 라벨은 힌트, 실제 보너스는 보수적으로”라는 원칙을 문서에 박는 것입니다. 지금 구조상 `QuestionSelector`가 정답 표시기처럼 보이면 안 되고, 반대로 이미 dormant인 효과를 확정 효과처럼 보여줘도 안 됩니다. 따라서 1차 구현은 `추천/주의/후속 선택 제안` 중심으로 두고, 실제 기계 효과 소비는 Phase 2 후반 또는 Phase 3에서 정합화하는 편이 안전합니다.]
