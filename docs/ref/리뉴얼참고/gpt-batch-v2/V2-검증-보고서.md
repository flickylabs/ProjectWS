## 2. CRITICAL 문제 (48/48 공통)

### 2-1. beatType 비표준 명명

- 템플릿에 `beatType` 허용값 목록이 명시되지 않음
- GPT가 라운드마다 다른 이름을 자유 생성:
  - **round-01**: `deny, hedge, partial, counter, emotion, confess, fatigue_irritation, fatigue_block, fatigue_counter, interjection_allow, interjection_block`
  - **round-02**: `deny, partial, counter, emotional, interject, rebuttal, escalate, clarify, irritate, block, retaliate`
  - **round-04**: `deny, hedge, partial, blame, emotional, confess` (evidence_hit/interjection 없음)
  - **round-07**: `justify, cornered, emotional, candid, irritated, blocked, counterattack, confused, mistaken, stuck, reframe, wavering, clarifying, counter, interject`
  - 등등 라운드마다 10~15종의 비표준 값 사용
- **스펙 허용값**: `deny, hedge, partial, blame, confession, evidence_hit` + interjection 관련
- **원인**: 템플릿에 beat 예시의 beatType이 `"deny"`로 나오지만, "이 값들만 사용하라"는 강제 없음

### 2-2. infoLevel 필드 누락 (48/48)

- interjection beat에 `infoLevel` 필드가 생성되지 않음
- 템플릿의 coverage matrix에 `emotional_only`, `detail_rebuttal` 등 4종이 언급되지만, beat JSON 예시에 `infoLevel` 필드가 포함되지 않음
- GPT가 이 필드를 생성할 근거가 없었음

### 2-3. truthEnvelope atomId V1 불일치 (spouse+family 24건)

- beats의 `truthEnvelope.allowAtomIds`/`forbidAtomIds`가 V1 v2-atoms의 실제 atom ID와 불일치
- GPT가 자체적으로 가상 ID를 생성 (`d1.care_overlap_claim`, `d1.breach_full_confess` 등)
- 템플릿에 "첨부된 v2-atoms.json에서 실제 atom ID를 검색해서 사용"이라는 강제가 부족
- friend+neighbor 24건은 V1 표준 형식과 일치 (이 라운드들은 GPT가 실제 첨부 파일을 참조한 것으로 보임)

## 3. HIGH 문제

### 3-1. conditions 하위 키 누락 (20/48)

- `emotionTiers`, `trustWindowBands`, `fatigueLevels` 3개가 누락된 beat 존재

- 템플릿 예시에 있지만 "모든 beat에 필수"라는 명시가 없음

### 3-2. interjection beat 부재 (16/48)

- `interject`, `interject_allow` 같은 비표준 타입은 있으나 `actionFamily: "interjection"` 패턴 미준수

### 3-3. round-01 allow/forbid 충돌 (4건)

- spouse-01~04의 confess beat에서 같은 atomId가 allow와 forbid에 동시 등록
- 런타임에서 모순 발생

### 4-1. 라운드별 품질 등급

| 라운드 | 대상 | 등급 | 핵심 이슈 |

| round-02 | spouse-05~08 | 중 | 대사 중복 27건 (surface/trap) |

| round-12 | neighbor-09~12 | 중하 | 조사오류, fatigue 템플릿, 대사 중복 |

### 4-2. 공통 대사 문제 패턴

- **fatigue beat 템플릿화**: 거의 모든 라운드에서 fatigue 3종(irritation/block/counter)이 dispute 불문 동일 문형. 캐릭터 개성 없음
- **interjection-fatigue 대사 복붙**: interjection beat의 line이 fatigue/counterattack beat와 100% 동일
- **d-X interjection a/b 동일 대사**: 양쪽 party가 같은 말을 함 (party 구분 불가)
- **조사 오류 패턴**: "~이 나오면" (받침 없는 글자 뒤 "이" 사용), "~오해고" ("오해로"), "X라는" ("X이라는")
- **surface/trap 대사 중복**: 동일 dispute에서 surface beat와 trap beat가 80~100% 동일

## 8. 프롬프트 고도화 제안

현재 템플릿에서 추가/강화해야 할 것:

1. **beatType 허용값 enum 명시**: `deny, hedge, partial, blame, confession, evidence_hit` (일반) + `interjection` 전용
2. **infoLevel 필드를 beat JSON 예시에 포함**: `"infoLevel": "emotional_only"` 등
3. **conditions 3필드 필수 명시**: "모든 beat에 반드시 포함"
4. **truthEnvelope atomId 강제**: "첨부된 v2-atoms.json의 claimAtoms에서 id 필드를 검색하여 정확히 사용. 새로운 ID를 만들지 마세요."
5. **fatigue beat 개별화 지시**: "dispute마다 다른 문장을 사용. 동일 문형을 dispute keyword만 바꿔 재사용하지 마세요."
6. **interjection a/b 차별화**: "같은 dispute의 party a와 party b는 반드시 다른 대사를 사용"
7. **대사 품질 체크리스트 추가**: 기존 12항목 -> 20항목으로 확대
