## 요청 A: Misconception 상태 전이 규칙 (M0→M4)

### 문제

DisputeV2에 `misconception` 프로파일(M0~M4 stages)이 있고,
BeatScript V2에 `misconceptionStates` 조건이 있지만,
**런타임에서 M 상태를 추적하고 전이하는 엔진이 없습니다.**

lieState(S0→S5)에는 `lieStateMachine.ts`의 `transitionLie()` + 트리거 시스템이 있는데,
misconception에는 이에 대응하는 것이 없습니다.

### 요청하는 설계

**1. M 상태 전이 트리거**
어떤 행동이 M 상태를 올리는가?

lieState 전이는 `질문 유형 → trigger → transitionLie()` 경로인데,
misconception은 "숨긴 진실을 자백"이 아니라 "오해가 풀리는 과정"이므로 트리거가 달라야 합니다.

예상:
- M0→M1: 해당 쟁점에 첫 질문
- M1→M2: 같은 쟁점 2~3번 추궁 (오해가 굳어지는 과정)
- M2→M3: `truthExitEvidence` 제시 or `trapSignal` 관련 질문
- M3→M4: `truthExitEvidence` 제시 + 관련 쟁점의 linkEdge 발동

이건 제 추측이고, **당신의 설계를 듣고 싶습니다.**

**2. store 확장**
- `misconceptionStateMap: Record<string, MisconceptionState>` (쟁점별)
- `transitionMisconception(disputeId, trigger)` — M 상태 전이 함수

**3. 기존 엔진 연결**
- `handleQuestion`에서 lieState 전이 시도와 별도로 misconception 전이도 시도
- `beatSelectorV2`의 `misconceptionState` 컨텍스트에 실 상태 전달
- `phase3LogCollector`에서 M4 도달 시 `recordDisprovedFakeIssue` 자동 호출

**4. spouse-01 d-3 + family-01 red_herring 기준 파일럿 시나리오**
- d-3: M0(외형상 의심) → M1(identity 공백) → M2(캡처 확신) → M3(상담동 맥락) → M4(해소)
- 각 전이에 필요한 조건을 구체적으로

### 출력
`misconception-engine-design.ts` — 타입 + 전이 규칙 + store 확장 + 통합 스니펫 + 파일럿 시나리오

## 요청 B: Free Question 분류기 + 제한 응답 엔진 설계

### 현재 상태

- `freeQuestionHooks` 데이터가 spouse-01(8개)과 family-01(7개)에 존재
- 각 hook에 `intentTag`, `allowedAtStates`, `answerEnvelope`, `refusalTemplates`가 있음
- 현재 `src/engine/llmFreeQuestion.ts`가 자유 질문을 처리하지만, V2 hooks와 연결되지 않음
- 확정 규칙: **자유 심문 단독으로 lieState +1 금지**

### 요청하는 설계

**1. 분류기 (intent classifier)**
플레이어 자유 입력 텍스트 → 가장 가까운 `intentTag` 매칭.

방식 선택지:
- A) 키워드 기반 (LLM 불필요, 빠름, 정확도 낮음)
- B) LLM 분류 (정확, 느림, 비용)
- C) 하이브리드 (키워드 1차 → 불확실하면 LLM)

**어떤 방식을 권장하시는지**, 그리고 그에 맞는 구현 설계를 해주세요.

**2. 제한 응답 생성**
분류된 intent가 `allowedAtStates` 안에 있으면:
- `answerEnvelope`의 허용 atom 범위 안에서 응답
- 응답 방식: beat 선택? 템플릿? 제한적 LLM?

분류된 intent가 범위 밖이면:
- `refusalTemplates`에서 랜덤 선택

**3. 효과**
- `hint` / `link reveal` / `readiness +0.5` / `trust 미세 변화` 정도만 허용
- **절대 금지: lieState 전이, misconception 전이**
- `phase3LogCollector`에 기록

**4. 파이프라인 위치**
현재 자유 질문은 비활성화 상태(`FreeQuestionInput` disabled).
활성화 조건: V2 사건 + S3 이상 + 턴 6 이상.

```
자유 입력 → intent 분류 → allowedAtStates 검사 → 허용이면 제한 응답, 아니면 거부
```

**5. 참고: 현재 llmFreeQuestion.ts**
첨부합니다. 이 파일의 구조를 이해한 뒤, V2 hooks 기반으로 교체하는 설계를 해주세요.

### 출력
`free-question-v2-design.ts` — 분류기 + 응답 생성 + 효과 + 파이프라인 + 통합 스니펫

## 요청 C: spouse-01 / family-01 Interjection beat 확장

### 문제

현재 interjection beat이 사건당 4개뿐 (allow 2 + block 2).
V2 끼어들기 시스템은 `quadrant × infoLevel` 조합에 따라 다른 대사가 필요합니다.

| infoLevel | 설명 | 필요 beat |
|-----------|------|----------|
| emotional_only | a_only/b_only — 감정적 끼어들기 | allow 2 + block 2 |
| detail_rebuttal | both_know — 구체 디테일 반박 | allow 2 + block 2 |
| misbelief_escalation | red_herring — 오해 강화 | allow 2 + block 1 |
| trap_signal | red_herring + knows — 함정 단서 | allow 2 + block 1 |

### 요청

**spouse-01**: 기존 4개 + 8개 추가 = 12개
**family-01**: 기존 4개 + 8개 추가 = 12개

각 beat는 기존 `beats-v2-full.json` 구조와 동일:
- `actionFamily: 'question'`이 아닌 interjection 전용 beat
- `tags: ['interjection', 'allow']` 또는 `['interjection', 'block']`
- `conditions.interjectionStates: ['allow_last_turn']` 또는 `['block_last_turn']`
- `infoLevel`에 맞는 대사 톤

### 참고 파일
- `ref-01-spouse01-structure-v2.json` — quadrant/misconception 참고
- `ref-02-family01-structure-v2.json` — quadrant/misconception 참고
- `ref-03-spouse01-beats-v2-full.json` — 기존 interjection beat 4개 (톤 레퍼런스)

### 출력
- `spouse-01-interjection-beats-v2.json` — 12개 beat
- `family-01-interjection-beats-v2.json` — 12개 beat
