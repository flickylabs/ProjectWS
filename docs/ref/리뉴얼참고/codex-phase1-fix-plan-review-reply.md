## Codex 회신

### Q1 (B4 수정 경로): [B 먼저 동의, 다만 구현 형태는 `stance=confess 시 usableInSubActions 전체 무시`보다 `S5 exact-slot rescue`에 가깝게 가는 편이 안전합니다. 현재 B4의 직접 원인은 후보 필터 단계에서 금액 atom이 빠지는 것이 맞고 (`src/engine/atomSelectionEngine.ts:90-109`), B를 먼저 넣으면 즉시 완화 효과가 큽니다. 다만 점수 규칙상 `motive_search`/`empathy_approach`는 여전히 감정·동기 atom을 우선하므로 (`src/types/claimV2.ts:216-233`), 단순 무시만으로는 exact amount atom이 항상 상위 2개에 들어온다는 보장이 없습니다. 따라서 1차는 엔진에서 `S5 + confess + selectedAtoms에 exact-support slot 없음`이면 amount/person/beneficiary/evidence slot이 있는 confess-safe atom 1개를 강제로 보강하는 방식이 좋고, 2차로 A 데이터 수정으로 `usableInSubActions` 자체를 정합화하는 순서가 맞습니다.]

### Q2 (실명 치환): [적절, 단 사건 데이터 name 참조만으로 끝내지 말고 치환 우선순위를 분리해야 합니다. `partyA.name` / `partyB.name`은 speaker 기준 `getJudgeReference()`로 치환하는 게 맞습니다 (`src/engine/llmSpeechGuide.ts:120-122`). 제3자 이름은 caseData `socialGraph[].name`을 참조해 감지하되 (`src/types/character.ts:54-102`), 기본 치환은 `그 사람` 또는 `그분`으로 두는 편이 안전합니다. 이유는 socialGraph에는 이름은 있어도 judgeRef가 항상 있는 구조가 아니기 때문입니다. 구현 시 주의점은 세 가지입니다. 1) 이름은 길이 내림차순으로 치환해 부분 매칭 충돌을 막을 것. 2) `[가-힣A-Za-z0-9_]` 경계를 써서 일반 단어 오탐을 줄일 것. 3) `씨/님`까지 붙은 변형을 같이 잡을 것. 가능하면 장기적으로는 사건 데이터 이름 목록 + claimPolicy slot의 `fullName → judgeRef` 매핑을 합친 dictionary를 쓰는 편이 더 정확합니다.]

### Q3 (P1-C 부작용): [조건부 필터 필요, blanket ignore는 부작용이 있습니다. 이유는 `usableInSubActions`를 무시해도 최종 선택은 태그 점수로 다시 갈리기 때문이고 (`src/engine/atomSelectionEngine.ts:111-146`), 동시에 subAction 전용으로 설계된 감정/관계 atom이 `fact_pursuit`에 섞일 수도 있기 때문입니다. 특히 `empathy_approach`는 `emotion/fear/shame/relationship/harm`를 강하게 우대하고 (`src/types/claimV2.ts:225-233`), 이때 fact/evidence 전용 confess atom까지 무차별 후보로 풀면 선택 다양성은 늘지만 응답 초점이 흐려질 수 있습니다. 따라서 권장 처리는 `필터 무시`가 아니라 `조건부 보강`입니다. 정상 selection을 먼저 하고, S5 confess인데 금액/실명/기관 같은 exact-support slot이 하나도 없을 때만 보강 atom을 1개 주입하거나, usableInSubActions mismatch를 hard reject 대신 score penalty로 낮추는 방식이 더 낫습니다.]

### Q4 (재검증 범위): [B, 근거는 두 가지입니다. 첫째, 이번 수정의 성공 판정 기준이 `CRITICAL 0, FAIL ≤10`이고, 현재 결함의 실질적 밀집 구간은 CRITICAL 13 + FAIL 32 = 45건이므로 이 버킷이 1차 수용 기준에 가장 직접 맞닿아 있습니다 (`tests/84-llm-quality-report.md`). 둘째, 84×3 전수는 비용이 크고 이번 변경이 명확히 Phase 1 결함 버킷을 겨냥하고 있으므로, 우선 45건으로 defect burn-down을 확인하는 게 실용적입니다. 다만 이번 P0/P1은 `llmDialogueResolver.ts` 후처리 전역 변경이 포함되므로 PASS 사건 회귀 위험이 0은 아닙니다. 그래서 공식 선택은 B로 두되, 실제 운영은 `45건 × 3회` 뒤에 PASS 사건 4~6건 정도를 sentinel smoke로 추가하는 것을 권장합니다.]

### 추가 의견: [가장 중요한 구현 리스크는 후처리 경로가 현재 두 군데라는 점입니다. 일반 parse 경로는 `parseLLMResponse()`에서 `fixMisdirectedAddress → enforceHonorifics → fixPostpositions`를 타고 (`src/engine/llmDialogueResolver.ts:923-926`), Blueprint V2 경로도 별도로 같은 체인을 직접 호출합니다 (`src/engine/llmDialogueResolver.ts:1576-1581`). P0/P1 후처리 가드를 한쪽에만 넣으면 재검증 결과가 왜곡됩니다. `postProcessNpcText(text, context)` 같은 공통 함수로 묶고, Truth Throttle / 금전 가드 / 어법 필터 / 조사 보정을 같은 순서로 태우는 편이 안전합니다. 추가로 P0의 `slotMaterial` 차단은 prompt에서만 막지 말고 atomPlan/slotSelections 로그에도 남겨서, 재검증 때 “금지 slot이 프롬프트에 안 들어갔는지”를 바로 확인할 수 있게 해두는 게 좋습니다.]
