## 프로젝트 배경

**Solomon**은 LLM 기반 법정 심문 게임입니다. 플레이어는 재판관으로서 양측 당사자(AI NPC)를 심문하여 진실을 밝힙니다.

- 엔진이 의미 결정, LLM은 연기만
- NPC는 lieState 상태 머신(S0 완전부정 → S5 시인)을 따름
- **ExecutableVerbalTell**: 캐릭터의 말버릇/언어 패턴 (lexicalHooks 포함)
- **BeatScript**: 쟁점별 deny/hedge/partial/blame/confession/evidence_hit 대사

## 재생성 목적

기존에 생성된 tells-beats의 **품질 문제를 수정**합니다:

1. **lexicalHook 미반영**: Tell에 정의된 핵심 말투 키워드가 beatScript 대사에 전혀 반영되지 않음 (반영률 0~22%, 기준 50% 이상)
2. **호칭 미반영**: 상대 호칭 시 callTerms.toPartner에 맞춰 "{{이름}} 씨" 미사용
3. **confession beat 톤 문제**: 자기 행위 명시 + "제 잘못/책임" 표현이 61% 누락
4. **일부 3인칭 서술체 혼입**: 재판관 앞 1인칭 발화여야 함

## 입력 파일 (첨부)

사건별로 다음 3개 파일을 첨부합니다:

1. **사건 원본 JSON** (`{{CASE_ID}}.json`) — 캐릭터 성격, speechStyle, archetype, callTerms 참조
2. **기존 tells-beats.ts** (`{{CASE_ID}}-tells-beats.ts`) — 구조 유지, **line만 재작성**
3. **v2-atoms.ts** (`{{CASE_ID}}-v2-atoms.ts`) — tellPool ID 참조용 (수정하지 않음)

## 출력

- `{{CASE_ID}}-tells-beats.ts` (기존과 동일 구조)
- `{{CASE_ID}}-tells-beats.json` (동일 내용 JSON 버전)

**구조는 100% 유지하고 각 beat의 `line` 필드만 재작성합니다.**

## 핵심 지시사항

### A. lexicalHook 반영 (최우선)

Tell에 정의된 `lexicalHooks` 배열의 키워드를 해당 party의 beatScript line에 **최소 50%** 자연스럽게 녹여주세요.

- Tell 1개에 hook 3개 → 최소 2개는 어딘가의 beat line에 등장해야 함
- party당 Tell 3개 × hook 3개 = 9개 중 최소 5개 이상 반영
- hook은 정확한 문구 그대로 또는 자연스러운 변형으로 포함 (예: "그냥 하소연" → "그건 그냥 하소연이었어요")
- 한 beat에 여러 hook을 넣어도 되고, 여러 beat에 분산해도 됨

### B. 존대/반말 기준

- **기본: 재판관 앞 존대체** (해요체 또는 합쇼체)
- archetype별 격식도:
  - `cold_logic`: 합쇼체 ("~입니다", "~습니다")
  - `confrontational`: 해요체 강한 ("~잖아요", "~거예요")
  - `avoidant`: 해요체 부드러운 ("~것 같아요", "~거든요")
  - `victim_cosplay`: 해요체 호소적 ("~했다고요", "~는 거예요")
- **"~다" 평서체(해라체) 사용 금지** — 어떤 archetype에서도 불가
- **감정 폭발 시 상대에게 직접 발화하는 경우만 반말 허용** (blame beat 등에서 제한적)

### C. 상대 호칭

- 상대를 지칭할 때 callTerms.toPartner에 맞춰 **"{{이름}} 씨"** 사용
- 예: toPartner가 "민수 씨"이면 beat line에서 "민수 씨가...", "민수 씨도..."
- 이름 없이 "상대방", "그쪽" 등은 최소화

### D. beatType별 톤 기준

#### deny (S0)
- 전면 부인: "~한 적 없다", "~가 아니다"
- 사실 인정 포함 금지

#### hedge (S1)
- 축소/맥락 전환: "~이긴 하지만", "~는 맞지만 그건..."
- 명시적 시인 금지

#### partial (S2)
- 부분 인정 + **반드시 "다만/그래도" 단서 포함**
- 전면 시인 금지

#### blame (S3)
- 인정 + **반드시 "~쪽도..." 상대 책임 전가 포함**
- 자기 책임만 인정하면 안 됨

#### confession (S4~S5)
- **필수 3요소**:
  1. 자기 행위 명시: "제가/내가 {{구체적 행위}}를..."
  2. 잘못/책임 인정: "제 잘못입니다" / "제 책임입니다" / "인정합니다" 중 하나 이상
  3. 회피 없는 종결: 상대 탓, 변명, 조건부("~이긴 한데...")로 끝나면 안 됨

#### evidence_hit (S2 이후)
- 증거 인식 표현: "~까지 나오면", "~부정 못 하겠다", "~보면 변명할 수 없네요"
- **confession과 다른 고유 톤 유지** — 자기 잘못 인정이 아니라 증거 앞 태도 변화
- confession 대사를 그대로 복붙하지 말 것

### E. 동업(partnership) 관계 특수성

- 계약, 지분, 성과 배분, 운영 권한 등 비즈니스 맥락에서의 자연스러운 표현
- 감정적 관계(가족, 친구)와 달리 "의무", "약속", "합의", "책임 범위" 등의 용어가 자연스럽게 등장
- 동업자 간 상호 존중("~대표", "~씨")과 비즈니스 갈등의 날카로움이 공존하는 톤

### F. beatScript 구조 유지

각 beat 객체에서 **line만 재작성**. 다른 필드는 기존 값 그대로 유지:
```json
{
  "id": "기존 유지",
  "caseId": "기존 유지",
  "party": "기존 유지",
  "disputeId": "기존 유지",
  "beatType": "기존 유지",
  "line": "← 이것만 재작성",
  "behaviorHint": "기존 유지",
  "applicableStates": "기존 유지",
  "afterEvidence": "기존 유지"
}
```

executableTells 구조도 기존 그대로 유지 (id, appliesWhen, lexicalHooks, sentenceShape, cadence, originalPattern 모두 유지).

## 3. 재생성 후 Claude Code가 할 작업

재생성된 파일을 GPT로부터 받은 후, 다음 순서로 적용한다:

### 3-1. 백업
```bash
# 기존 파일 백업 (사건별로)
cp {{CASE_ID}}-tells-beats.ts {{CASE_ID}}-tells-beats.ts.bak
cp {{CASE_ID}}-tells-beats.json {{CASE_ID}}-tells-beats.json.bak
```

### 3-2. 파일 교체
- 새로 받은 tells-beats.ts/.json으로 기존 파일 교체
- TS 파일의 export 이름이 기존과 동일한지 확인

### 3-3. 정규화 스크립트 실행
- JSON/TS 포맷 검증
- ID 패턴 정규식 매칭 (콜론 구분자, 하이픈 없는 caseId 등)
- beatType 허용값 6종 검증
- applicableStates 값 검증

### 3-4. tellPool 매칭 재검증
- v2-atoms.ts의 `tellPool` 배열에 있는 Tell ID가 새 tells-beats.ts의 `executableTells`에 모두 존재하는지 확인
- ID가 1글자라도 다르면 런타임 에러이므로 정확히 일치해야 함

### 3-5. 의미 검증 재실행
- lexicalHook 반영률 재측정 (목표: 50% 이상)
- confession beat 3요소 점검

- evidence_hit vs confession 중복 여부 점검
- 호칭 기준 점검 (callTerms.toPartner 반영)

### 3-6. 결과 리포트

- 검증 실패 항목 (있을 경우)

## 5. 주의사항

### 5-1. V2 배치와 절대 혼합 금지
- tells-beats 재생성 GPT 스레드에서 v2-atoms나 v3-game-loop-data를 동시에 요청하지 않는다
- V2 배치(84건 9종 생성)는 별도 계획에 따라 진행

### 5-2. tellPool ID 보존
- v2-atoms.ts에 정의된 `tellPool` 배열의 Tell ID는 절대 변경 불가
- 재생성된 tells-beats의 `executableTells` ID가 기존과 100% 동일해야 함
- 새 Tell을 추가하거나 기존 Tell을 삭제하지 않는다

### 5-3. beat 구조 완전 유지
- `id`, `caseId`, `party`, `disputeId`, `beatType`, `applicableStates`, `afterEvidence`, `behaviorHint` — 모두 기존 값 그대로
- **오직 `line` 필드만 재작성**
- beat 개수 변경 금지 (추가/삭제 불가)

### 5-4. 검증 실패 시 대응
- hook 반영률이 50% 미달이면 GPT에 해당 beat를 지목하여 재요청
- confession 3요소 누락 시 해당 beat만 재요청
- 구조 변경이 발견되면 기존 .bak 파일에서 구조 복원 후 line만 반영
