## 재생성 사유

### 공통 문제
1. **대사 템플릿 복붙**: 3개 dispute에 걸쳐 동일 문장이 반복되고 있음
2. **존대/반말 불규칙**: partyB의 존대/반말이 일관되지 않음

### 추가 지시

- **각 dispute의 대사를 개별화해주세요.** 3개 dispute가 동일 문장으로 시작하거나 동일 구조를 반복하면 안 됩니다.
- **partyB의 존대/반말을 일관되게 유지해주세요.** archetype에 맞는 화체 하나로 통일.

## 입력 파일 (첨부)

사건별로 다음 3개 파일을 첨부합니다:

1. **사건 원본 JSON** (`사건원본_{{CASE_ID}}.json`) — 캐릭터 성격, speechStyle, archetype, callTerms 참조
2. **기존 tells-beats.ts** (`{{CASE_ID}}-tells-beats.ts`) — 구조 유지, **line만 재작성**
3. **v2-atoms.ts** (`{{CASE_ID}}-v2-atoms.ts`) — tellPool ID 참조용 (수정하지 않음)

## 출력

- `{{CASE_ID}}-tells-beats.ts` (기존과 동일 구조)
- `{{CASE_ID}}-tells-beats.json` (동일 내용 JSON 버전)

**구조는 100% 유지하고 각 beat의 `line` 필드만 재작성합니다.**

---

## 필수 규칙: ID 네이밍

> 기존 beat의 id, caseId, party, disputeId, beatType, applicableStates, afterEvidence, behaviorHint는 **일절 변경하지 마세요.** line만 재작성합니다.

### ID 공통 규칙 (참조용)
- 구분자는 반드시 콜론(`:`)
- caseId 접두사에 하이픈 없음: `friend06` (O) / `friend-06` (X)
- Tell ID: `{{CASE}}:{{PARTY}}:tell:{{TELL_NAME}}`
- Beat ID: `{{CASE}}:beat:{{PARTY}}:{{DISPUTE}}:{{BEAT_TYPE}}`

## 핵심 지시사항

### A. lexicalHook 반영 (최우선)

Tell에 정의된 `lexicalHooks` 배열의 키워드를 해당 party의 beatScript line에 **최소 50%** 자연스럽게 녹여주세요.

- Tell 1개에 hook 3개 → 최소 2개는 어딘가의 beat line에 등장해야 함
- party당 Tell 3개 × hook 3개 = 9개 중 최소 5개 이상 반영
- hook은 정확한 문구 그대로 또는 자연스러운 변형으로 포함
- 한 beat에 여러 hook을 넣어도 되고, 여러 beat에 분산해도 됨

### B. 존대/반말 기준

- **기본: 재판관 앞 존대체** (해요체 또는 합쇼체)
- archetype별 격식도:
  - `cold_logic`: 합쇼체 ("~입니다", "~습니다")
  - `confrontational`: 해요체 강한 ("~잖아요", "~거예요")
  - `avoidant`: 해요체 부드러운 ("~것 같아요", "~거든요")
  - `victim_cosplay`: 해요체 호소적 ("~했다고요", "~는 거예요")
- **"~다" 평서체(해라체) 사용 금지** — 어떤 archetype에서도 불가
- **감정 폭발 시 상대에게 직접 발화하는 경우만 반말 허용** (blame beat 등에서 제한적)

### C. 상대 호칭

- 상대를 지칭할 때 callTerms.toPartner에 맞춰 **"{{이름}} 씨"** 사용
- 이름 없이 "상대방", "그쪽" 등은 최소화

### D. beatType별 톤 기준

#### deny (S0)
- 전면 부인: "~한 적 없다", "~가 아니다"
- 사실 인정 포함 금지

#### hedge (S1)
- 축소/맥락 전환: "~이긴 하지만", "~는 맞지만 그건..."
- 명시적 시인 금지

#### partial (S2)
- 부분 인정 + **반드시 "다만/그래도" 단서 포함**
- 전면 시인 금지

#### blame (S3)
- 인정 + **반드시 "~쪽도..." 상대 책임 전가 포함**
- 자기 책임만 인정하면 안 됨

#### confession (S4~S5)
- **필수 3요소**:
  1. 자기 행위 명시: "제가/내가 {{구체적 행위}}를..."
  2. 잘못/책임 인정: "제 잘못입니다" / "제 책임입니다" / "인정합니다" 중 하나 이상
  3. 회피 없는 종결: 상대 탓, 변명, 조건부("~이긴 한데...")로 끝나면 안 됨

#### evidence_hit (S2 이후)
- 증거 인식 표현: "~까지 나오면", "~부정 못 하겠다", "~보면 변명할 수 없네요"
- **confession과 다른 고유 톤 유지** — 자기 잘못 인정이 아니라 증거 앞 태도 변화
- confession 대사를 그대로 복붙하지 말 것

### E. 친구 관계 특수성

- 오랜 친분, 공유된 경험, 감정적 유대감 등 친구 관계 맥락에서의 자연스러운 표현
- 비즈니스 관계(동업)와 달리 "믿었는데", "우리 사이에", "그동안", "친구라서" 등의 표현이 자연스럽게 등장
- 친구 간 친밀함과 배신감/서운함이 공존하는 톤

### F. dispute별 대사 개별화 (이번 재생성 핵심)

- **3개 dispute가 동일 문장으로 시작하거나 동일 구조를 반복하면 안 됩니다.**
- 같은 beatType이라도 dispute마다 다른 어휘, 다른 문장 구조, 다른 감정 톤을 사용해주세요.
- 예시:
  - dispute-1 deny: "그런 적 없어요."
  - dispute-2 deny: "오해가 있으신 것 같습니다."
  - dispute-3 deny: "제가 왜 그랬겠습니까."
- 특히 deny, hedge, confession에서 복붙이 가장 심하므로 집중 개별화해주세요.

### G. beatScript 구조 유지

각 beat 객체에서 **line만 재작성**. 다른 필드는 기존 값 그대로 유지:
```json
{
  "id": "기존 유지",
  "caseId": "기존 유지",
  "party": "기존 유지",
  "disputeId": "기존 유지",
  "beatType": "기존 유지",
  "line": "← 이것만 재작성",
  "behaviorHint": "기존 유지",
  "applicableStates": "기존 유지",
  "afterEvidence": "기존 유지"
}
```

executableTells 구조도 기존 그대로 유지 (id, appliesWhen, lexicalHooks, sentenceShape, cadence, originalPattern 모두 유지).
