# 추가 요청 5차 — V2 배치 템플릿 업데이트 + Free Question 키워드 확장 방안

## 상황

4차 산출물(misconception 전이 규칙 + free question V2 + interjection beat 12개) 전량 채택, 구현 완료.
현재 V2 파이프라인 상태:

| 엔진 | 상태 |
|------|------|
| misconceptionEngine V2 | ✅ 구현 완료 — trigger 9종 + belief mode별 threshold |
| freeQuestionV2 | ✅ 구현 완료 — deterministic-first 분류기 |
| interjection beat 확장 | ✅ spouse-01 74개, family-01 68개 (각 12 interjection) |
| Phase 6/Result 프롬프트 | ✅ 구현 완료 — Phase3PromptBridgeV2 기반 |
| beatSelectorV2 | ✅ 기존 대로 |
| 엔진↔게임루프 연결 | 🔧 다음 단계 |

파일럿 2건(spouse-01, family-01)은 완전 동작 상태.
**다음 단계는 C스레드에서 82사건 V2 데이터 배치 생성**입니다.

이번 요청은 그 배치 전에 **확정해야 할 2가지**입니다.

## 요청 A: V2 추가배치 템플릿 업데이트

### 문제

현재 `00-V2-추가배치-템플릿.md`의 interjection 섹션이 구버전입니다:
```
### Interjection beats (필수)
- allow 2개 + block 2개 (최소)
```

4차에서 확정된 실제 구조는 **사건당 12개**이며, quadrant × infoLevel 축을 커버합니다:
- `emotional_only` (4개) — 순수 감정 끼어들기
- `detail_rebuttal` (4개) — 구체 디테일 반박
- `misbelief_escalation` (2~4개) — red_herring 쟁점 전용, beliefMode 차등
- `trap_signal` (0~2개) — 가짜 단서 감지 신호 (red_herring 있을 때만)

### 요청하는 산출물

**템플릿의 interjection 섹션을 교체할 수 있는 정확한 텍스트**.

아래 내용을 반영해 주세요:

1. **beat 수**: 12개/사건 (최소 10, 최대 14)
2. **infoLevel 분포**:
   - emotional_only: 4개 (allow 2 + block 2)
   - detail_rebuttal: 4개 (allow 2 + block 2)
   - misbelief_escalation: 2~4개 (red_herring 쟁점 있을 때만, allow+block)
   - trap_signal: 0~2개 (red_herring 쟁점 있을 때만, allow만 또는 allow+block)
3. **필수 필드**:
   - `actionFamily: "interjection"` (기존 beat의 `"question"` 대신)
   - `beliefMode`: red_herring/shared_misconception beat에만 (knows/suspects/misbelief/weaponizes)
   - conditions 내: `interjectionStates`, `misconceptionStates`, `trapStates`
4. **ID 패턴** 예시:
   - `{{CASE}}:beat_v2:{{PARTY}}:{{DISPUTE}}:surface:mid:interjection:{{CHOICE}}:{{INDEX}}`
   - 예: `family02:beat_v2:b:d-3:surface:mid:interjection:allow:01`
5. **antiRepeatGroup 패턴**: `interjection.{{CHOICE}}.{{PARTY}}`
6. **tags 규칙**: `["interjection", "allow"|"block", "event_lane", INFOLEVEL_TAG]`
   - infoLevel tag: `emotional_only` / `detail_rebuttal` / `misbelief_escalation` / `trap_signal`
   - red_herring 관련이면 `"red_herring"` 추가
7. **coverageKey 패턴** 예시도 함께

### 체크리스트 업데이트

현재 체크리스트의 이 줄:
```
- [ ] beats 총 40~60개 + fatigue 3종/dispute + interjection allow2+block2
```
→ 아래로 교체:
```
- [ ] beats 총 50~74개 + fatigue 3종/dispute + interjection 12개(infoLevel 4종)
```

## 요청 B: Free Question V2 키워드 확장 전략

### 문제

현재 freeQuestionV2 분류기에 `DISPUTE_ALIAS` 사전이 있습니다:
```ts
const DISPUTE_ALIAS: Record<string, string[]> = {
  'd-1': ['280만원', '1800만원', '비밀 송금', '선이체', '예약금', '간병비'],
  'd-2': ['휴대폰', '폰', '캡처', '새벽', '60만원', '지연', '방치'],
  'd-3': ['외도', '최민정', '골목', '상담동', '수첩', '상속', '한 줄'],
  ...
}
```

이 사전은 **spouse-01/family-01 전용**이라 82사건 확장 시 동작하지 않습니다.
또한 `INTENT_LEXICON`은 범용 한국어 패턴이라 상대적으로 재사용 가능하지만,
사건별 고유 키워드(이름, 장소, 금액, 증거명 등)는 반드시 사건별로 달라야 합니다.

### 선택지

**방안 A**: structure-v2 스키마에 `disputeAliases` 필드를 dispute 레벨로 추가
```json
{
  "id": "d-1",
  "name": "비밀 송금",
  "disputeAliases": ["280만원", "1800만원", "선이체", "예약금"],
  ...
}
```
- 장점: dispute마다 자연스러운 위치, GPT 배치에서 dispute 설명 보면서 바로 작성 가능
- 단점: 기존 DisputeV2 타입 확장 필요 (작은 변경)

**방안 B**: freeQuestionHook에 `triggerKeywords` 필드 추가
```json
{
  "id": "fq-spouse01-01",
  "intentTag": "identity_check",
  "triggerKeywords": ["누구", "최민정", "정체"],
  ...
}
```
- 장점: hook 단위로 정밀 제어, intent별 키워드 분리 가능
- 단점: hook 8개 × 키워드 3~5개 = 사건당 24~40개 키워드를 GPT가 채워야 함

**방안 C**: A + B 혼합 — dispute에 alias, hook에 intent 특화 keyword 양쪽 배치

### 요청하는 설계 결정

1. **어떤 방안이 "작성 단가 1분 이하" 원칙에 맞는가?**
2. **선택한 방안의 스키마 patch** (JSON 필드 정의 + 예시 1~2개)
3. **V2 추가배치 템플릿에 추가할 관련 작성 규칙** (1~2줄)
4. **`INTENT_LEXICON`은 범용으로 유지해도 되는가?** — 82사건에서 intent 패턴이 크게 달라지는 경우가 있다면 intent도 사건별로 확장해야 할 수 있음
