# 솔로몬 사건 데이터 재작업 요청 — 가족(family) 07~12

> 이 프롬프트는 **self-contained**입니다. 이전 세션을 참고하지 마세요.
> 첨부된 원본 JSON 6건과 현재결과를 기반으로 작업해주세요.

---

## 1. 프로젝트 개요

**솔로몬**은 AI NPC 2명(당사자 A/B)이 분쟁하고, 플레이어가 재판관으로서 심문·증거조사를 통해 진실을 밝히는 법정 추리 게임입니다.

각 사건에는 5개 쟁점(d-1~d-5)이 있고, 각 당사자는 쟁점별로 거짓말 상태(S0~S5)를 가집니다.
플레이어의 행동(질문, 증거제시, 공감, 동기탐색 등)에 따라 거짓말이 점진적으로 붕괴됩니다.

---

## 2. 재작업 대상 5종 — 정확한 스펙

### 2.1 actionAffinity.bestActionHint / worstActionReaction

**용도**: 플레이어가 특정 쟁점에서 특정 행동을 선택했을 때, 왜 그것이 효과적인지(또는 비효과적인지) 설명하는 텍스트. 게임 내 힌트 시스템과 AI 프롬프트에서 사용.

**스펙**:
- 쟁점당 1개씩 (사건당 5개 × 6건 = 30개)
- bestActionHint: "이 쟁점에서 이 행동이 왜 효과적인지" — 사건의 구체적 인물명, 증거, 갈등 내용을 포함
- worstActionReaction: "이 행동을 하면 NPC가 왜 더 방어적이 되는지" — 동일하게 사건 맥락 반영

**좋은 예** (spouse-01, d-1):
```json
"bestActionHint": "체면과 평판 계산을 말로 꺼내게 하면, 숨긴 이유가 스스로 드러나며 방어 논리가 약해진다.",
"worstActionReaction": "사실만 몰아붙이면 단어 정의나 일정표 해석에 매달리며 본론을 더 늦춘다."
```
→ 쟁점의 핵심(체면/평판 동기)과 NPC 반응 패턴이 구체적.

**나쁜 예** (현재 S4~S7에서 반복되는 문구):
```json
"bestActionHint": "원본 로그와 문서 대면이 가장 빠르게 해석 싸움을 끝내고 방어를 흔든다.",
"worstActionReaction": "공감만 건네면 피해 서사나 정당화만 길어지고 핵심 검증이 늦어진다."
```
→ 어떤 사건, 어떤 쟁점에나 붙일 수 있는 범용 문구. 이런 출력은 거부됩니다.

**검증 기준**: 30개 bestActionHint 중 **동일 문구 2개 이상 불가**. 30개 worstActionReaction도 마찬가지.

### 2.2 witnessSpeechSample

**용도**: AI가 증인 NPC의 대사를 생성할 때 few-shot 예시로 참조하는 실제 대화체 샘플.

**스펙**:
- 증인당 2~3개 샘플 (사건당 3 증인 × 6건 = 18명)
- 각 샘플은 **해당 사건의 구체적 사실**을 언급하는 실제 대화체
- 증인의 말투, 성격, 입장이 드러나야 함
- canState 텍스트를 그대로 복사하면 안 됨

**핵심 규칙**: speechSample은 canState를 **패러프레이즈하면 안 됩니다.** canState에 없는 새로운 상황, 감정, 에피소드를 포함한 **완전히 다른 문장**이어야 합니다.

- canState = "이 증인이 알고 있는 사실 목록" (서술체, 3인칭)
- speechSample = "이 증인이 법정에서 실제로 말하는 대사" (대화체, 1인칭, 감정 포함)

**좋은 예**:
```
canState: "추석 연휴에 가족이 간병 인력을 알아보던 분위기가 있었다."
speechSample: "그때 제가 옆에서 봤는데, 어머니가 갑자기 '이제 돈 좀 모아야 해'라면서 한숨을 쉬시더라고요. 그 분위기가 심상치 않았어요."
```
→ canState의 "간병 인력" 키워드를 반복하지 않고, **증인만의 관찰/감정/구체적 순간**을 담은 새로운 문장.

**나쁜 예 1** (복사):
```
canState: "계약서에 '원상복구' 조항이 있었다"
speechSample: "계약서에 '원상복구' 조항이 있었고, 입주 전 사진도 남겨두었습니다"
→ FAIL: canState를 거의 그대로 옮김
```

**나쁜 예 2** (패러프레이즈):
```
canState: "명절 직후 공급처들 사이에 공장 매각 소문이 돌기 시작한 시점을 안다."
speechSample: "제가 아는 건 시점입니다. 명절 직후 갑자기 같은 내용의 문의가 몰렸고..."
→ FAIL: canState의 핵심 구문("명절 직후", "시점")을 그대로 가져와 대화체로 바꾼 것 뿐
```

**올바른 작성법**:
1. canState를 먼저 읽지 말고, 원본 사건 JSON에서 증인의 입장/관계/성격을 파악
2. "이 증인이 법정에서 자기 입으로 말한다면?" 상상
3. canState에 없는 **구체적 에피소드**(시간, 장소, 감정 반응, 대화 장면)를 만들어서 작성
4. canState에 등장하는 금액, 날짜, 고유명사를 speechSample에서 반복하지 마세요. 같은 사건이라도 증인은 **자기만의 관찰 포인트**가 있습니다.
5. 작성 후 canState와 비교하여 키워드 겹침이 30% 미만인지 확인

**검증 기준**: speechSample과 canState의 텍스트 유사도가 50% 이상이면 FAIL. 패러프레이즈도 FAIL. **canState와 동일한 사건/숫자/인물명을 그대로 언급하는 것도 FAIL.**

### 2.3 truthCategory

**용도**: 각 진실(truth)의 성격을 분류하여, 게임 엔진이 해금 순서와 난이도를 조절하는 데 사용.

**스펙**:
- truth당 1개 카테고리 (사건당 5개 × 6건 = 30개)
- **5종 카테고리 중 택1**:
  - `core_fact` — 객관적으로 검증 가능한 사실 (계약서, 금액, 날짜, 장소)
  - `subjective_claim` — 당사자의 주관적 주장/해석 ("상대가 먼저 시작했다")
  - `emotional_context` — 감정적 배경/심리 상태 ("무시당한 느낌", "배신감")
  - `hidden_motive` — 숨겨진 동기 (체면, 보복, 경제적 이득, 관계 유지)
  - `circumstantial` — 정황/간접 증거 (타이밍, 목격, 제3자 전언)

**검증 기준**:
- 6건(30개 truth) 전체에서 한 카테고리가 50% 이상 차지하면 FAIL
- 전체 최소 3종 이상 사용
- **각 사건(5개 truth) 내에서 최소 3종 카테고리 사용** — 한 사건이 core_fact 4개 + circumstantial 1개처럼 2종만 쓰면 FAIL
- 모든 사건에는 감정이나 동기가 포함된 truth가 최소 1개는 있으므로, core_fact만으로 채우지 마세요

### 2.4 witnessBudgets.forbidden / uncertain

**용도**: 각 증인이 법정에서 말해선 안 되는 것(forbidden)과 확신 없이 말하는 것(uncertain).
AI 프롬프트에서 증인의 증언 범위를 제한하는 데 사용.

**스펙**:
- 증인당 forbidden 2~3개, uncertain 2개
- **사건의 구체적 내용**을 참조해야 함 (인물명, 문서명, 사건 등)

**좋은 예** (spouse-01, tp-1):
```json
"uncertain": [
  "280만원 송금의 정확한 시각과 명목은 직접 보지 못했다.",
  "최민정을 만난 밤 골목 접촉의 실제 목적은 전해 들은 수준이다."
],
"forbidden": [
  "세린의 휴대폰 무단 열람과 제3자 캡처 공유의 구체적 경위",
  "정우성 계좌를 거친 150만원의 실제 금융 흐름",
  "지석이 실제로 어떤 설명을 하려다 말았는지에 대한 단정"
]
```
→ 사건 고유의 인물명과 구체적 사실이 포함됨.

**나쁜 예** (현재 S4~S7, 36건 동일):
```
"직접 확인하지 못한 사적 대화 전문이나 개인기기 내부 경위를 사실처럼 말하는 것"
```
→ 어떤 사건에나 적용 가능한 범용 문구.

**검증 기준**: 18명의 forbidden 중 동일 문구 3개 이상 불가.

### 2.5 personalityTags

**용도**: 각 당사자의 성격 유형을 태그로 분류. 게임 엔진이 NPC 반응 패턴과 상성 계수를 결정하는 데 사용.

**스펙**:
- 당사자당 4개 태그 (사건당 2명 × 6건 = 12명)
- 아래 태그 풀에서 선택 (사건 맥락에 맞게):

```
confrontational, passive_aggressive, gaslighting, denial_heavy,
blame_shifting, overly_cooperative, martyr_complex, detail_obsessed,
emotionally_volatile, calculated_calm, trust_broken, grudge_holding,
conflict_avoidant, authority_challenging, victim_identity,
manipulative, shame_sensitive, face_sensitive, fairness_obsessed,
cold_logical, selective_quote, timeline_padding, counter_attack,
avoidant, relationship_preserving, retaliation_sensitive, privacy_sensitive
```

**검증 기준**: 12명 중 동일 조합 2쌍 이상 불가. `face_sensitive`가 전체의 50% 이상이면 FAIL.

## 3. 출력 형식

### 출력 파일명: `재작업결과_family_07to12.json`

첨부된 `현재결과_family_07to12.json`과 **동일한 구조**로 출력하되, 위 5개 섹션만 교체:
- `actionAffinity` — bestActionHint, worstActionReaction만 교체 (actionScores, bestAction, worstAction은 유지)
- `witnessSpeechSample` — 전체 교체
- `truthCategory` — 전체 교체
- `witnessBudgets` — forbidden, uncertain만 교체 (name, canState 유지)
- `personalityTags` — 전체 교체

나머지 5종(truthPolicies, contentTags, optimalPath, narrativeExpansion, actionScores)은 **그대로 유지**.

## 4. 첨부 파일 목록

| 파일 | 용도 |
|------|------|
| **`레퍼런스_workplace-07.json`** | **품질 기준 샘플 — 이 파일의 수준과 동일하게 작성할 것** |
| `family-NN.json` (6건) | 원본 사건 데이터 — 인물명/쟁점/증거/진실 확인용 |
| `현재결과_family_07to12.json` | 현재 결과물 — 구조는 유지, 위 5종만 교체 |

### 레퍼런스 파일 사용법
`레퍼런스_workplace-07.json`은 Session 1에서 가장 잘 작성된 결과물입니다.
이 파일의 모든 데이터 타입(9종)이 **출력 품질의 기준선**입니다.

특히 주목할 점:
- `actionAffinity.bestActionHint` — 사건의 구체적 인물/동기/증거를 언급하는 힌트
- `witnessSpeechSample` — canState와 다른, 실제 대화체 톤의 증언 샘플
- `witnessBudgets.forbidden` — 사건 고유 인물명과 구체적 문서를 언급하는 제한 사항
- `personalityTags` — 다양한 태그 조합 (face_sensitive에만 의존하지 않음)
- `truthCategory` — fact 외에 다양한 카테고리 (이 레퍼런스는 fact/motive만 사용했지만, 재작업에서는 5종 카테고리 체계를 사용)

**이 레퍼런스보다 낮은 품질의 출력은 거부됩니다.**
