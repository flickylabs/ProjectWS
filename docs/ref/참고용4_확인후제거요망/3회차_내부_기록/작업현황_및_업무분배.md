## 4. 다른 에이전트에게 전달할 작업 상세

### A1. truthIds 매핑표 설계

**목적**: 쟁점(dispute)별, 거짓말 상태(lieState)별로 "이번 턴에 말해도 되는 진실"과 "절대 누설하면 안 되는 진실"을 구조화.

**참고 자료**:
- 사건 JSON 샘플 3개: `에이전트_참고자료/cases/` (spouse-01, workplace-01, friend-01)
- 각 사건의 `truthTable` 배열: `quadrant`(both_know/a_only/b_only/neither_knows), `fact`, `isTrue`
- 각 사건의 `lieConfig`: `disputeId`, `initialState`, `transitions`, `collapseViaTrust`, `lieMotive`

**요청 산출물**:
```
사건별 매핑표:
  dispute d-1 (쟁점명):
    S0: allowed=[t-3], forbidden=[t-1, t-2, t-4, t-5]
    S1: allowed=[t-3, t-6], forbidden=[t-1, t-2, t-4]
    S2: allowed=[t-3, t-6, t-7], forbidden=[t-1, t-2]
    ...
    S5: allowed=[전체], forbidden=[]
```

**핵심 원칙**:
- S0~S1: 표면적 사실만 허용. 핵심 비밀(anchorTruth)은 금지
- S2~S3: 일부 배경 사실 허용. 핵심은 여전히 금지
- S4: 감정적 사실 허용. 핵심 근처까지 접근 가능
- S5: 전체 허용
- `collapseViaTrust=true`인 쟁점은 S4에서도 핵심 접근 가능 (신뢰 루트)
- `lieMotive`에 따라 forbidden 범위가 달라짐 (예: shame_avoidance는 수치 관련 사실이 더 강하게 금지)

**형식**: 사건 3개(spouse-01, workplace-01, friend-01)에 대해 작성. 나머지 81개에 적용할 패턴도 함께 제시.

### A4. User Message 템플릿 검토/보강

**목적**: v3 JSON의 `userMessageTemplates`를 실제 사건 데이터에 대입해보고, 부자연스럽거나 부족한 부분을 다듬기.

**참고 자료**:
- v3 JSON의 `userMessageTemplates` 섹션 전체
- 사건 JSON 샘플 3개 (disputes, evidence, truthTable)
- 현재 `llmDialogueResolver.ts`의 `buildUserPrompt()` 함수 (server/llmDialogueResolver.ts)

**현재 userMessageTemplates 구조** (v3 JSON):
```
question: { fact_pursuit, motive_search, empathy_approach }
evidence_present
evidence_investigate: { request_original, check_metadata, restore_context, verify_source, check_edits, question_acquisition }
trust_action: { confidential_protection, separation, emotional_stabilization, retaliation_check }
free_question_classifier
free_question_responder
witness_call
```

**요청 사항**:
1. 각 템플릿에 spouse-01 사건의 실제 값을 대입한 예시 1개씩 작성
2. 대입 결과가 부자연스러우면 템플릿 수정안 제시
3. 특히 `evidence_investigate` 6종의 `{investigationResult}` 부분이 실제 사건 JSON의 investigationResults와 자연스럽게 연결되는지 확인
4. `trust_action`의 `{focusedDisputeId}` + `{judgeQuestion}` 조합이 자연스러운지 확인

### A2. 증인 witnessBudget 초안 (1차 완료 후)

**목적**: 253명 증인의 "말할 수 있는 것 / 불확실한 것 / 말하면 안 되는 것"을 구조화.

**참고 자료**:
- 사건 JSON의 `duo.socialGraph` 배열 (각 증인의 `knowledgeScope`, `bias`, `distortionRisk`, `witnessProfile`)
- `witness_base` 블록의 `{witnessBudget}` 변수 형식

**요청 산출물** (사건 3개에 대해):
```json
{
  "tp-1": {
    "canState": ["세린이 9월 초 전화로 남편 외도 의심을 얘기했다", "지석이 추석 전 돈 걱정을 했다"],
    "uncertain": ["송금 시점은 정확히 모른다", "모텔 건은 전해 들었다"],
    "forbidden": ["간병비 진실은 모른다", "지석의 실제 행선지"]
  }
}
```

**핵심 원칙**:
- `witnessedDirectly: true`인 증인은 `canState`가 더 풍부
- `bias: pro_a/pro_b`인 증인은 한쪽에 유리한 해석을 `canState`에 포함하되, 반대쪽 유리한 사실은 `uncertain`으로
- `distortionRisk: intentional/strategic`인 증인은 특정 사실을 `forbidden`에 포함 (자기에게 불리한 것)
- `knowledgeScope` 밖의 모든 사실은 자동으로 `forbidden`

### A3. Eval 케이스 설계 (1차 완료 후)

**목적**: 프롬프트 변경 후 AI 응답 품질을 측정할 기준선 마련.

**요청 산출물**: 사건 3개 × 시나리오 5개 = 15개 eval 케이스

각 케이스:
```
사건: spouse-01
액션: fact_pursuit, d-1, target=a, lieState=S1
기대 stance: deny 또는 hedge
기대 responseMode: answer_only
금지: t-1(간병비 진실) 언급
체크: d-2~d-5 쟁점 이름/금액 누출 없음
좋은 응답 예시: "재판관님, 280만원은 9월 15일에 보냈습니다. 급한 일이었습니다."
나쁜 응답 예시: "사실 그 돈은 어머니 간병비..." (S1에서 핵심 진실 노출)
```

**시나리오 5종**:
1. fact_pursuit + S0~S1 (부정 상태에서 사실 추궁)
2. empathy_approach + S2~S3 (약간 흔들린 상태에서 공감)
3. evidence_present + hard 증거 (강한 증거 앞 반응)
4. separation + S3 (분리심문에서 보복 걱정 감소)
5. confidential_protection + S4 (비공개 보호에서 솔직해지는지)
