📬 공지 & 우편 시스템 v1.0

관리자 → 플레이어 커뮤니케이션 파이프라인 — 공지사항 실시간 표시 & 보상 우편 발송

문서 범위: 이 문서는 공지(Notice)와 우편(Mail) 시스템의 DB 스키마, API 엔드포인트, 프론트엔드 UI 컴포넌트, 운영 시나리오를 상세히 다룹니다. 서버 아키텍처 전반은 서버 & DB, 관리자 도구는 WebAdmin 문서를 참조하세요.

1. 공지 시스템 (Notice System)

1.1 시스템 개요

공지 시스템은 관리자가 WebAdmin 대시보드에서 공지를 작성하면, 게임 클라이언트의 TitleScreen에서 실시간으로 표시되는 단방향 커뮤니케이션 시스템입니다. 공지는 서버의 SQLite 데이터베이스에 저장되며, 클라이언트는 TitleScreen 진입 시 GET /api/notices API를 호출하여 활성 공지 목록을 가져옵니다.

핵심 설계 원칙
  • 즉시 반영: 관리자가 공지를 작성하는 즉시 클라이언트에서 조회 가능 (별도 배포 불필요)
  • 기간 기반 자동 관리: start_date / end_date로 노출 기간을 지정하면 서버가 자동으로 필터링
  • 타입별 시각적 구분: 4가지 공지 타입에 따라 색상 배지가 자동 적용
  • 우선순위 + 고정: 긴급 공지는 항상 최상단에 표시

1.3 우선순위 (Priority)

값레벨표시 방식정렬 가중치적합한 상황
0 일반 기본 스타일 최하위 일반 안내, 이벤트 공지
1 중요 강조 테두리 중간 업데이트 안내, 중요 변경사항
2 긴급 빨간 강조 + 상단 배치 최상위 긴급 점검, 치명적 버그 안내
정렬 규칙: 공지 목록은 항상 pinned DESC → priority DESC → created_at DESC 순서로 정렬됩니다. 즉, 고정 공지가 최상단, 그 다음 우선순위가 높은 순, 마지막으로 최신순으로 표시됩니다.

1.4 고정 공지 (Pinned)

고정 공지 동작 방식
  • pinned = 1 (true): 공지 목록 최상단에 항상 고정 표시. 📌 아이콘이 제목 앞에 렌더링됩니다.
  • pinned = 0 (false): 일반 공지로 우선순위와 생성일 기준 정렬됩니다.
  • 복수 고정: 여러 공지를 동시에 고정할 수 있으며, 고정 공지 간에는 priority → created_at 순으로 정렬됩니다.
  • WebAdmin 조작: 공지 편집 화면에서 "상단 고정" 토글 스위치로 제어합니다.
DB 스키마 — notices 테이블
CREATE TABLE notices (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  title       TEXT    NOT NULL,              -- 공지 제목
  content     TEXT    NOT NULL,              -- 공지 본문 (HTML 또는 텍스트)
  type        TEXT    NOT NULL DEFAULT 'info', -- 'info' | 'event' | 'update' | 'maintenance'
  priority    INTEGER NOT NULL DEFAULT 0,    -- 0=일반, 1=중요, 2=긴급
  pinned      INTEGER NOT NULL DEFAULT 0,    -- 0=일반, 1=상단고정
  is_active   INTEGER NOT NULL DEFAULT 1,    -- 0=비활성, 1=활성
  start_date  TEXT    DEFAULT NULL,          -- 노출 시작일 (ISO 8601, null=즉시)
  end_date    TEXT    DEFAULT NULL,          -- 노출 종료일 (ISO 8601, null=무기한)
  created_at  TEXT    DEFAULT (datetime('now')),
  updated_at  TEXT    DEFAULT (datetime('now'))
);

1.5 노출 기간 (Exposure Period)

start_dateend_date동작사용 시나리오
NULL NULL 즉시 노출, 무기한 유지 영구 안내 (이용약관 등)
2026-04-01 NULL 4월 1일부터 무기한 노출 예정된 변경사항 안내
NULL 2026-04-30 즉시 노출, 4월 30일에 자동 종료 기간 한정 이벤트 공지
2026-04-01 2026-04-30 4월 1일~30일만 노출 시즌 이벤트 공지
서버측 기간 필터링 로직
// server/routes/notices.js — 활성 공지 조회
const now = new Date().toISOString();

const notices = db.prepare(`
  SELECT * FROM notices
  WHERE is_active = 1
    AND (start_date IS NULL OR start_date <= ?)
    AND (end_date   IS NULL OR end_date   >= ?)
  ORDER BY pinned DESC, priority DESC, created_at DESC
`).all(now, now);

2. 우편 시스템 (Mail System)

2.1 시스템 개요

우편 시스템은 관리자가 플레이어에게 우편을 발송하고, 선택적으로 보상을 첨부할 수 있는 시스템입니다. 공지 시스템과 달리 플레이어별 읽음/수령 상태를 개별 추적하며, 보상이 첨부된 경우 플레이어가 명시적으로 "보상 받기" 버튼을 클릭해야 수령이 완료됩니다.

공지 vs 우편 비교
특성공지 (Notice)우편 (Mail)
방향관리자 → 전체 플레이어관리자 → 전체 또는 특정 플레이어
보상 첨부불가가능 (coin, item, title)

2.3 보상 시스템 (Reward System)

reward_typereward_amountreward_data설명예시
null null null 보상 없음 — 단순 안내 우편 환영 메시지
'coin' 500 null 게임 머니 지급 사과 보상: 코인 500개
'item' 1 {"itemId":"hint_scroll","name":"힌트 두루마리"} 아이템 지급 이벤트 보상 아이템
'title' 1 {"titleId":"early_bird","name":"얼리버드"} 칭호 지급 사전 예약 칭호
보상 처리 흐름
플레이어 동작                     서버 처리                          결과
─────────────                ─────────────                    ─────────
1. 우편 목록 진입     →       GET /api/mail/player/:id         우편 목록 표시
2. 우편 클릭 (열람)   →       POST /api/mail/read              is_read = true
3. "보상 받기" 클릭   →       POST /api/mail/claim             중복 체크 후 보상 반환
                              ├─ reward_claimed 이미 true?     → 409 Conflict
                              ├─ reward_type이 null?           → 400 Bad Request
                              └─ 정상                          → 200 + 보상 데이터
4. 클라이언트 처리    ←       { reward_type, reward_amount }   로컬 저장소 반영
중복 수령 방지: mail_read_status 테이블의 UNIQUE(mail_id, player_id) 제약과 reward_claimed 플래그로 이중 보호됩니다. 동일 우편에 대한 보상 수령 요청이 중복 도착하더라도 서버에서 반드시 차단합니다.

2.4 읽음/수령 상태 (Read/Claim Status)

DB 스키마 — mail_read_status 테이블
CREATE TABLE mail_read_status (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  mail_id         INTEGER NOT NULL,              -- mail.id 참조
  player_id       TEXT    NOT NULL,              -- 플레이어 고유 ID
  is_read         INTEGER NOT NULL DEFAULT 0,    -- 0=미읽음, 1=읽음
  reward_claimed  INTEGER NOT NULL DEFAULT 0,    -- 0=미수령, 1=수령완료
  read_at         TEXT    DEFAULT NULL,          -- 읽은 시각
  claimed_at      TEXT    DEFAULT NULL,          -- 수령 시각
  created_at      TEXT    DEFAULT (datetime('now')),
  UNIQUE(mail_id, player_id)                     -- 중복 방지 제약
);
필드초기값변경 시점변경 API되돌림 가능
is_read 0 플레이어가 우편 상세 뷰를 열 때 POST /api/mail/read 불가 (단방향)
reward_claimed 0 플레이어가 "보상 받기" 버튼 클릭 시 POST /api/mail/claim 불가 (단방향)
UPSERT 패턴 — 읽음 처리
// POST /api/mail/read — 우편 읽음 처리
db.prepare(`
  INSERT INTO mail_read_status (mail_id, player_id, is_read, read_at)
  VALUES (?, ?, 1, datetime('now'))
  ON CONFLICT(mail_id, player_id) DO UPDATE SET
    is_read = 1,
    read_at = COALESCE(mail_read_status.read_at, datetime('now'))
`).run(mailId, playerId);

// ON CONFLICT → UPSERT: 이미 레코드가 있으면 UPDATE
// COALESCE: 최초 읽은 시각 보존 (이후 재열람 시 시각 변경 안 함)
보상 수령 처리
// POST /api/mail/claim — 보상 수령
router.post('/claim', (req, res) => {
  const { mailId, playerId } = req.body;

  // 1. 우편 존재 및 보상 확인
  const mail = db.prepare('SELECT * FROM mail WHERE id = ?').get(mailId);
  if (!mail) return res.status(404).json({ error: '우편을 찾을 수 없습니다' });
  if (!mail.reward_type) return res.status(400).json({ error: '보상이 없는 우편입니다' });

  // 2. 이미 수령했는지 확인
  const status = db.prepare(
    'SELECT * FROM mail_read_status WHERE mail_id = ? AND player_id = ?'
  ).get(mailId, playerId);

  if (status?.reward_claimed) {
    return res.status(409).json({ error: '이미 수령한 보상입니다' });
  }

  // 3. 수령 처리 (UPSERT)
  db.prepare(`
    INSERT INTO mail_read_status (mail_id, player_id, is_read, reward_claimed, read_at, claimed_at)
    VALUES (?, ?, 1, 1, datetime('now'), datetime('now'))
    ON CONFLICT(mail_id, player_id) DO UPDATE SET
      reward_claimed = 1,
      claimed_at = datetime('now')
  `).run(mailId, playerId);

  // 4. 보상 데이터 반환
  res.json({
    success: true,
    reward: {
      type: mail.reward_type,
      amount: mail.reward_amount,
      data: mail.reward_data ? JSON.parse(mail.reward_data) : null,
    }
  });
});

2.5 만료 시스템 (Expiration)

expires_at동작사용 시나리오
NULL 무기한 유지 — 플레이어가 삭제하지 않는 한 영구 보관 환영 우편, 중요 안내
'2026-04-30T23:59:59Z' 해당 시각 이후 API 응답에서 자동 제외 이벤트 보상 (기간 한정)
주의: 만료된 우편의 미수령 보상은 소멸됩니다. 운영 시 만료일을 넉넉하게 설정하거나, 중요한 보상은 expires_at = NULL로 설정하는 것을 권장합니다.
서버측 만료 필터링
// GET /api/mail/player/:playerId — 플레이어 우편 조회
const mails = db.prepare(`
  SELECT m.*,
         mrs.is_read,
         mrs.reward_claimed,
         mrs.read_at,
         mrs.claimed_at
  FROM mail m
  LEFT JOIN mail_read_status mrs
    ON m.id = mrs.mail_id AND mrs.player_id = ?
  WHERE m.is_active = 1
    AND (m.expires_at IS NULL OR m.expires_at > datetime('now'))
    AND (m.target_type = 'all' OR (m.target_type = 'specific' AND m.target_id = ?))
  ORDER BY m.created_at DESC
`).all(playerId, playerId);

상단 바 버튼 구현
// TitleScreen.tsx — 상단 바 버튼
<div style={{ display: 'flex', gap: '0.5rem' }}>
  {/* 공지 버튼 */}
  <button onClick={() => setShowNotice(true)} disabled={!isServerConnected}>
    📢
  </button>

  {/* 우편 버튼 (카운트 배지 포함) */}
  <button
    onClick={() => setShowMail(true)}
    disabled={!isServerConnected}
    style={{ position: 'relative' }}
  >
    📨
    {unreadCount > 0 && <span className="badge">{unreadCount}</span>}
  </button>
</div>

{/* 모달 렌더링 */}
{showNotice && <NoticePanel onClose={() => setShowNotice(false)} />}
{showMail && <MailInbox playerId={playerId} onClose={() => setShowMail(false)} />}

서버 연결 확인 코드
// src/services/api.ts — Health Check
export const healthApi = {
  check: async (): Promise<boolean> => {
    try {
      const response = await fetch(`${API_BASE}/api/health`, {
        signal: AbortSignal.timeout(3000),  // 3초 타임아웃
      });
      return response.ok;
    } catch {
      return false;
    }
  },
};

// TitleScreen.tsx — 시작 시 연결 확인
useEffect(() => {
  healthApi.check().then(connected => {
    setIsServerConnected(connected);
    if (connected) {
      // 병렬로 공지/우편/프롬프트 로드
      Promise.allSettled([
        noticeApi.getActive(),
        mailApi.getUnreadCount(playerId),
        promptApi.getActive(),
        playerApi.sync(localProfile),
      ]);
    }
  });
}, []);

3.3 오프라인 모드

오프라인 시 동작 정책
기능서버 연결 시오프라인 시비고
공지 시스템 📢 버튼 활성, 클릭 시 공지 목록 표시 📢 버튼 비활성 (dimmed), 클릭 불가 서버 데이터 의존
우편 시스템 📨 버튼 활성, 카운트 배지 표시 📨 버튼 비활성 (dimmed), 배지 숨김 서버 데이터 의존
게임 플레이 정상 플레이 + AI 심문 가능 정상 플레이 (AI 심문 제외) 로컬 데이터로 플레이 가능
AI 프롬프트 서버에서 최신 프롬프트 로드 로컬 캐시 프롬프트 사용 캐시 없으면 AI 기능 비활성
플레이어 동기화 localStorage → 서버 자동 동기화 동기화 스킵, localStorage만 사용 다음 연결 시 자동 동기화
설계 원칙 — 오프라인 우선 (Offline-First): 솔로몬은 싱글 플레이어 게임이므로, 서버 연결이 불가능하더라도 핵심 게임 플레이(사건 선택, 증거 조사, 심문, 판결)는 로컬 데이터만으로 정상 진행됩니다. 공지/우편/AI 심문 등 서버 의존 기능만 비활성화되며, 이는 게임 플레이에 치명적이지 않습니다.

playerApi.sync() 구현
// src/services/api.ts
export const playerApi = {
  sync: async (profile: PlayerProfile): Promise<PlayerProfile> => {
    const response = await fetch(`${API_BASE}/api/players/sync`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        player_id: profile.id,
        nickname: profile.name,
        level: profile.level,
        total_cases: profile.totalCases,
        correct_verdicts: profile.correctVerdicts,
      }),
    });
    return response.json();
  },
};

// 동기화 시점:
// 1. 앱 시작 시 (TitleScreen mount)
// 2. 사건 완료 시 (통계 변경)
// 3. 레벨업 시

기능동기화 필요 데이터용도
우편 시스템 player_id 플레이어별 우편 조회/읽음/수령 처리
WebAdmin 대시보드 전체 프로필 플레이어 목록 조회, 통계 확인
특정 플레이어 우편 player_id + nickname WebAdmin에서 특정 플레이어 우편 발송 시 대상 검색
